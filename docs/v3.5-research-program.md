# Genorah v3.5 Research Program

10 empirical tracks validating numeric choices in v3.5.0–3. Each track has owner, protocol, duration, success metric, and output artifact in this directory.

Runs in parallel with shipping. Defaults ship as "provisional"; recalibration from track results upgrades them in-place.

## R1 — Judge Calibration Empirics

**Question:** Optimal few-shot N? κ threshold? Drift rate?
**Protocol:**
1. Build 50-section golden set spanning 25 archetypes × 2 beats (HOOK/PEAK). Three-human consensus panel scores each on both axes (Design/234, UX/120).
2. Sweep few-shot N ∈ {0, 1, 3, 5}. Measure RMSE vs consensus per N.
3. Run 2-judge κ per session across 30 days. Measure drift (ΔRMSE/day).
4. Cross-validate: 10 sections held out.
**Success:** RMSE ≤ 12pt at chosen N. Agreement κ ≥ 0.7 stable ≥ 2 weeks.
**Output:** `r1-judge-calibration.md` → few-shot N, κ floor, recal cadence defaults.

## R2 — Persona Prompt Validation

**Question:** Do the 6 UX personas produce useful signal?
**Protocol:**
1. Instrument 30 sections with known-good and known-injected UX issues (touch target violations, jargon inserts, above-fold CTA removal, etc.).
2. Run each persona on each section. Measure recall (flagged real issues) + precision (didn't flag false) per persona.
3. Inter-persona redundancy (Jaccard overlap on findings).
**Success:** Recall ≥ 0.8, precision ≥ 0.7 per kept persona. Redundancy ≤ 0.4 between retained pair.
**Output:** `r2-persona-validation.md` → kept/cut personas, prompt tuning recommendations.

## R3 — Pareto-N Sweep

**Question:** Optimal variant count for HOOK/PEAK/CLOSE beats?
**Protocol:**
1. Generate N ∈ {1, 3, 5, 7, 10} variants on 20 HOOK/PEAK sections.
2. Measure final Pareto-selected score vs total token cost.
3. Fit diminishing-returns curve per beat.
**Success:** Elbow identified; report score/$ per N.
**Output:** `r3-pareto-sweep.md` → recommended N per budget tier (lean/standard/max).

## R4 — Critic Cycle Sweep

**Question:** Optimal critic cycle cap?
**Protocol:**
1. Run 0/1/2/3/4 critic cycles on 20 sections.
2. Measure score trajectory per cycle, oscillation (score drop after improvement), regression rate.
**Success:** Cap chosen minimizes oscillation while capturing ≥ 90% of total available improvement.
**Output:** `r4-critic-cycles.md` → cap value + early-termination rule tuning.

## R5 — UX Floor Calibration per Archetype

**Question:** What UX score actually means "pass" per archetype?
**Protocol:**
1. Build 5 sections × 5 UX scores × 25 archetypes = 625 sections (or sample).
2. Survey 20 working designers: pass/fail judgment per section.
3. Fit ROC curve per archetype; select threshold at Youden's J max.
**Success:** Per-archetype floor with ≥ 85% human agreement.
**Output:** `r5-ux-floor-matrix.md` → archetype → UX floor, Brutalist vs Editorial differential validated.

## R6 — Reference Library Sizing

**Question:** How many curated references saturate RAG benefit?
**Protocol:**
1. Build RAG with sizes {50, 200, 500, 1000} curated SOTD sections.
2. Measure variant-baseline lift (pre-Pareto) across 50 trials per size.
3. Archetype coverage uniformity (variance of references per archetype).
**Success:** Saturation curve + gap-filling priority list.
**Output:** `r6-rag-sizing.md` → library size + archetype gap map.

## R7 — Budget Tier Token Verification

**Question:** Do lean / standard / max token estimates hold?
**Protocol:**
1. Run full pipeline on 5 real projects per tier.
2. Measure actual tokens + time + final quality delta (Design + UX scores).
3. Compute $/quality-point per tier.
**Success:** Verified or revised tier budget envelopes.
**Output:** `r7-budget-tiers.md` → validated tier defaults.

## R8 — Archetype Weight Matrix

**Question:** Empirical axis weights per archetype vs current guesses?
**Protocol:**
1. Panel survey (20 designers): rank axis importance (Color, Typography, Motion, Creative Courage, UX, Conversion, A11y, Performance) per archetype.
2. Aggregate via Borda count.
3. Compare to current 1.0–1.4× weights; flag significant deltas.
**Success:** Empirically-derived weight matrix shipped; current guesses verified or replaced.
**Output:** `r8-archetype-weights.md` → weight matrix.

## R9 — Context Fabric Efficacy

**Question:** Does Fabric actually reduce rework?
**Protocol:**
1. A/B real projects with Fabric mechanisms toggled on/off (L2, L4, L5, L6, L7 each toggled independently).
2. Measure:
   - Decision-coherence rate (same decision made twice?)
   - Rework % (sections rebuilt after initial ship)
   - User-rework-after-delivery % (user edits in first 30 days post-ship)
3. N = 20 projects minimum; effect size with CI.
**Success:** Each mechanism either validated (effect size > 0.3 Cohen's d) or cut.
**Output:** `r9-fabric-efficacy.md` → go/no-go per mechanism.

## R10 — Synthetic-User Signal-to-Noise

**Question:** False-pos / false-neg rate vs real users?
**Protocol:**
1. Run synthetic + real-user test (usertesting.com or similar) on 10 sections.
2. Measure agreement, FP/FN per persona.
**Success:** Per-persona confidence weights calibrated; low-SNR personas cut or reweighted.
**Output:** `r10-synthetic-snr.md` → persona confidence weights.

## Infrastructure

- **`research-runner` agent** — spawns experiments, manages variants, writes to journal + calibration store.
- **Variant Archive** (`~/.claude/genorah/variants/`) — every variant generated archived with full metadata for retrospective analysis.
- **Metrics Warehouse** (`~/.claude/genorah/warehouse.duckdb`) — per-project METRICS + journals + trajectories aggregated for cross-project queries.
- **A/B Toggle** — `quality.experiments` block in `.claude/genorah.local.md` flips mechanisms per project for controlled comparison.
- **Research Dashboard** — `/gen:dashboard --tab=research` live experiment tracking (sample sizes, significance, effect sizes).
- **Quarterly Recalibration Ritual** — `/gen:recalibrate` command: re-runs R1 + R5 against updated golden set + new variant archive; publishes diff vs previous thresholds; prompts approval to update shipping defaults.

## Shipping Policy

1. All v3.5.0–2 numeric defaults ship marked **PROVISIONAL** in `.planning/` output.
2. v3.5.3 incorporates R1–R10 results into shipping defaults, removing PROVISIONAL tags.
3. Quarterly recalibration ritual re-validates; drift > 10% triggers review.
