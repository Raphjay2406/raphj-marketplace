**context-rot-prevention.md** (~250-350 lines):

This is a protocol document (not a YAML-frontmatter agent). It defines the complete 6-layer defense system that the build-orchestrator and other agents reference. Format: structured markdown with clear sections for each layer.

**6-Layer Defense System:**

Layer 0 -- Pre-Commit DNA Compliance Hook (Zero context cost):
- Mechanism: Shell script runs OUTSIDE context window via PreToolUse hook on Bash tool
- Location: `.claude-plugin/hooks/dna-compliance-check.sh`
- What it checks: greps committed files for anti-slop violations
- Violations detected: raw hex values not in DNA, Tailwind default classes (rounded-lg, shadow-md, text-blue-500), system fonts (Inter, Arial, sans-serif), banned micro-copy phrases
- Consequence: commit REJECTED. Hard block, no override
- Context cost: ZERO (shell script, not in LLM context)

Layer 1 -- CONTEXT.md Anchoring (Low context cost):
- Mechanism: Single source of truth file, rewritten after every wave (~80-100 lines)
- Purpose: Provides quick orientation for any agent (especially orchestrator returning after session break)
- Structure (define exact format):
  ```
  # Modulo Context
  Last updated: [ISO date] | Wave: [N] completed | Session: [N]

  ## DNA Identity (~20 lines)
  Archetype: [name]
  Display: [font] | Body: [font] | Mono: [font]
  Colors: [all 12 tokens with hex values, 2 lines]
  Spacing: [5-level scale, 1 line]
  Radius: [system, 1 line]
  Shadows: [system, 1 line]
  Motion: [easing, stagger, enter directions, 2 lines]
  FORBIDDEN: [patterns, 1 line]
  Signature: [element + params, 1 line]

  ## Build State (~20 lines)
  | Section | Wave | Status | Beat | Layout | Background |
  [compact table of all sections]

  ## Creative Direction Notes (~15 lines)
  [CD's observations from last wave: strengths, drift, push opportunities]

  ## Emotional Arc Position (~5 lines)
  Completed: [beat sequence so far]
  Current: [what's being built]
  Upcoming: [what's next]

  ## Feedback Loop (~10 lines)
  REPLICATE: [patterns that worked]
  AVOID: [patterns that lost points]
  DESIGN-SYSTEM: [shared components added]

  ## Next Wave Instructions (~10 lines)
  Wave [N+1]: [section names]
  Plans: [PLAN.md paths]
  Session: [continue | recommend new session]
  First action: [what to do]
  ```
- Ownership: Orchestrator writes DNA Identity, Build State, Emotional Arc, Feedback Loop, Next Wave. CD writes Creative Direction Notes
- Rewrite protocol: FULL REWRITE after every wave (not append). Creative notes trimmed to last wave only. Build state table includes ALL sections but in compact format
- Size management: Target 80-100 lines. If exceeding 100 lines, compress: abbreviate section statuses, trim older feedback, summarize rather than list

Layer 2 -- Pre-Extracted Spawn Prompts (Amortized cost):
- Mechanism: Orchestrator pre-extracts ALL context into each builder's spawn prompt
- Effect: Builders get fresh context windows with exactly what they need
- Budget: ~150 lines spawn prompt + ~200 lines PLAN.md = ~350 lines total
- What's pre-extracted: Full DNA identity, beat assignment + parameters, adjacent sections, layout patterns used, shared components, content for this section, quality rules, lessons learned
- Key insight: The extraction cost is paid ONCE by the orchestrator. Each builder's context window starts clean

Layer 3 -- Canary Checks (Minimal cost):
- Mechanism: 5 questions answered from memory after every wave, verified against CONTEXT.md
- Reference: See canary-check.md protocol for full details
- Cost: ~50 tokens per check (5 answers + scoring)
- When: MANDATORY after every wave, CANNOT be skipped

Layer 4 -- Session Boundaries (Zero cost):
- Mechanism: Policy-based session management
- Soft boundary (every 2 waves): Orchestrator force-writes CONTEXT.md, suggests "consider starting a new session" to user. Build CAN continue if user chooses
- Hard boundary (turn 31+): Mandatory session save. Orchestrator writes CONTEXT.md with explicit resume instructions. Message to user: "Session boundary reached. Start a new session and run /modulo:execute to continue."
- Canary-triggered boundary: If canary check scores 0-2, treat as hard boundary regardless of wave count or turn count
- Resume protocol: New session reads CONTEXT.md, picks up from Next Wave Instructions

Layer 5 -- Baked-In Rules in Agent Files (Zero cost):
- Mechanism: Critical rules embedded directly in agent markdown system prompts
- Examples: Beat parameter table in section-builder, anti-slop quick check in builder, creative review checklist in CD
- Effect: Rules are part of the agent's system prompt -- present from the first token, never forgotten
- Tradeoff: Agent files are longer (~300-400 lines), but rules are always available

**Warning Signs of Context Rot:**
List specific observable indicators that canary checks or reviewers should watch for:
- Canary check failures (2+ wrong answers)
- Stray hex values appearing in built code (not from DNA)
- Tailwind defaults appearing: shadow-md, rounded-lg, gap-4, text-gray-500
- System font names appearing: Inter, Arial, Helvetica, sans-serif
- Beat parameters not matching constraints (BREATHE with 30% whitespace)
- Signature element missing from sections
- Archetype forbidden patterns appearing
- Generic copy despite CONTENT.md having specific text
- All sections using centered layouts (compositional flatness)
- Animations all being fade-in-up (motion monotony)

**When Rot Is Detected:**
1. Flag the drift with specific evidence (what drifted, how)
2. Present to user with options: fix in-place, rebuild section, or continue anyway
3. If continuing: add drift observation to CONTEXT.md feedback loop
4. If 2+ canary failures: recommend session boundary (hard recommend, user can override)

**canary-check.md** (~100-150 lines):

Protocol document defining the canary check procedure that the orchestrator runs after every wave.

**Purpose:**
Canary checks detect context rot BEFORE it produces bad output. By testing the orchestrator's recall of key project identity and state facts, we can catch degradation early.

**When to Run:**
- After EVERY wave completion (mandatory, cannot be skipped)
- Before constructing next wave's spawn prompts
- If manually triggered by user

**The 5 Questions:**

DNA Recall (tests identity drift):
1. "What is our project's display font?" (answer: exact font name)
2. "What is our accent-1 hex value?" (answer: exact hex)
3. "What patterns are forbidden by our archetype?" (answer: list forbidden patterns)

State Recall (tests context drift):
4. "What layout patterns have been used in completed sections?" (answer: list of patterns)
5. "What beat type is assigned to the next section to build?" (answer: beat name)

**Procedure:**
1. Orchestrator answers ALL 5 questions from memory (no file reads)
2. Orchestrator reads CONTEXT.md to verify answers
3. Score each answer: correct or incorrect
4. Record score in CONTEXT.md build state

**Scoring & Consequences:**

| Score | Status | Action |
|-------|--------|--------|
| 5/5 | HEALTHY | Continue normally. Log "Canary: HEALTHY" in CONTEXT.md |
| 3-4/5 | DEGRADING | Re-read CONTEXT.md carefully. Log "Canary: DEGRADING" in CONTEXT.md. Add extra DNA tokens to next spawn prompts. Continue with caution |
| 0-2/5 | ROT_DETECTED | TRIGGER SESSION BOUNDARY. Log "Canary: ROT_DETECTED" in CONTEXT.md. Save full CONTEXT.md with resume instructions. Message user: "Context rot detected ([score]/5). Strongly recommend starting a new session." |

**Question Substitution:**
Questions can be adapted per project. The KEY PRINCIPLE is:
- 3 questions test DNA/identity recall (things that should never change)
- 2 questions test recent state recall (things that change every wave)
- Questions should have ONE correct answer (not ambiguous)

**Integration with Build Orchestrator:**
The orchestrator runs this protocol as step 11 in the wave execution protocol (after coherence checkpoint, before next wave prep). The canary check is NOT a separate agent -- it's a protocol the orchestrator follows.

**Anti-Gaming:**
The orchestrator must answer from MEMORY first, then verify. If it reads CONTEXT.md before answering, the check is meaningless. The protocol explicitly states: "Answer ALL 5 questions BEFORE reading any files."
