<objective>
Define the quality-reviewer and polisher agents as the two-part quality enforcement pipeline.

Purpose: Quality enforcement in v6.1.0 was a single agent that both diagnosed and suggested fixes. In v2.0, this splits into two agents: the quality-reviewer diagnoses problems through goal-backward verification and anti-slop scoring, then produces structured GAP-FIX.md files; the polisher reads those files and makes targeted fixes with minimal context. This separation means the reviewer can be thorough (reading many files) while the polisher stays lean (reading only the gap + the code).

Output: Two agent markdown files that handle the review-then-fix cycle.
</objective>

**Anti-Slop 35-Point Scoring:**
Embed the full 7-category scoring system. The exact point distribution and category weights will be defined in Phase 1 (anti-slop gate skill). For now, define the structure:

| Category | Points | What It Measures |
|----------|--------|-----------------|
| Colors | /5 | DNA token usage, no raw hex, color harmony, expressive tokens used |
| Typography | /5 | DNA fonts, type scale hierarchy, no system defaults |
| Layout | /5 | Compositional diversity, no adjacent repeats, responsive |
| Depth & Polish | /5 | Shadows, textures, micro-interactions, hover states |
| Motion | /5 | Animation variety, prefers-reduced-motion, scroll integration |
| Creative Courage | /5 | Tension moments, wow factor, archetype boldness |
| UX Intelligence | /5 | Micro-copy quality, CTA specificity, content hierarchy |

Score < 25/35 = FAIL (hard gate)
Score 25-27 = PASS
Score 28-29 = STRONG
Score 30-32 = SOTD-READY

Penalties: Missing signature element (-3), archetype forbidden pattern (-5), no creative tension in PEAK beat (-5).

**polisher.md** (~150-200 lines):
Frontmatter: name: polisher, description (applies targeted fixes from GAP-FIX.md files, reads minimal context, makes atomic changes, updates SUMMARY.md), tools: Read, Write, Edit, Bash, Grep, Glob, model: inherit, maxTurns: 20.

System prompt must define:

**Input contract (minimal context):**
- Reads: ONE GAP-FIX.md file (specifies what to fix)
- Reads: The specific code files listed in GAP-FIX.md
- Reads: DESIGN-DNA.md (for token reference during fixes)
- Does NOT read: STATE.md, BRAINSTORM.md, CONTENT.md, MASTER-PLAN.md, REFERENCES.md, or any skill files

**Output contract:**
- Modifies: Code files specified in GAP-FIX.md
- Updates: Section's SUMMARY.md (marks gaps as fixed, updates anti-slop self-check)
- Writes: Polish confirmation in GAP-FIX.md (status: RESOLVED)

**Two-tier operation:**

Light Polish (triggered automatically after every section build -- handled by section-builder, not polisher):
- This is NOT the polisher's job. Light polish is embedded in the section-builder's final stage.
- The polisher is only invoked when GAP-FIX.md exists.

Deep Polish (triggered by GAP-FIX.md from quality-reviewer or creative-director):
- Read GAP-FIX.md to understand what needs fixing
- For each gap: read the specific file, make the targeted fix, verify the fix
- Atomic: one fix at a time, verify before moving to next
- After all fixes: update GAP-FIX.md status to RESOLVED, update SUMMARY.md

**Fix protocol:**
1. Read GAP-FIX.md -- understand all gaps and their severity
2. Sort by severity: critical first, then major, then minor
3. For each gap:
   a. Read the specific code file(s)
   b. Apply the fix described in the "Fix" field
   c. Verify the fix addresses the "Truth" that failed
   d. If fix requires DNA tokens: reference DESIGN-DNA.md
   e. Commit atomically (one fix per commit)
4. After all fixes: re-check anti-slop items that were below threshold
5. Update GAP-FIX.md: add "Resolution" field to each gap
6. Update SUMMARY.md: mark gaps as fixed, update scores

**Rules:**
- NEVER change code that isn't listed in GAP-FIX.md files (scope discipline)
- NEVER "improve" adjacent code while fixing a gap (scope creep)
- If a fix requires changes beyond the listed files: document it as a new gap, don't expand scope
- If a fix conflicts with another fix: resolve in order of severity
- Always verify fixes don't break existing functionality (run dev server if needed)
