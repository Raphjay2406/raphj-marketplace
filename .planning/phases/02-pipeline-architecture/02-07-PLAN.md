**agent-memory-system.md** (~200-300 lines):

Protocol document defining the 3-layer agent memory architecture. This is NOT a YAML agent -- it's a reference document that other agents follow.

**Layer 1: Living Context File (CONTEXT.md)**

Purpose: Short-term project memory -- current state, recent decisions, immediate next steps.
Location: `.planning/genorah/CONTEXT.md`
Size target: 80-100 lines (hard ceiling: 120 lines)
Update cadence: Rewritten (not appended) after every wave
Ownership: Split -- orchestrator writes build state, CD writes creative notes (see context-rot-prevention.md)

Read by: build-orchestrator (primary consumer), creative-director (reads creative notes section)
Written by: build-orchestrator (after every wave), creative-director (creative notes section)

Growth management:
- Full rewrite prevents unbounded growth
- Creative notes: only LAST wave's observations kept (previous waves' notes are superseded)
- Build state table: all sections shown but compact format (one row per section)
- Feedback loop: only last 2 waves of lessons kept (older lessons are distilled or dropped)
- If over 100 lines after rewrite: compress feedback to top 3 items, abbreviate section statuses

When to create: After Design DNA generation (Phase 1 output). Initial CONTEXT.md contains DNA identity + empty build state + empty creative notes.
When to retire: After all waves complete and quality review passes. Final CONTEXT.md serves as project record.

**Layer 2: Growing Design System (DESIGN-SYSTEM.md)**

Purpose: Medium-term memory -- accumulated reusable components and shared patterns.
Location: `.planning/genorah/DESIGN-SYSTEM.md`
Size target: Grows organically, no hard ceiling (this is a reference document, not an LLM context file)
Update cadence: After every wave, orchestrator appends new entries from builder SUMMARY.md files

Written by: build-orchestrator (aggregates from builder SUMMARY.md files)
Read by: build-orchestrator (to populate "Shared Components Available" in spawn prompts)

Format:
```markdown
# Design System

## Shared Components

### Wave 0-1 (Foundation)
| Component | Path | Usage | Added |
|-----------|------|-------|-------|
| Navbar | src/components/nav.tsx | Global navigation | Wave 0 |
| Footer | src/components/footer.tsx | Global footer | Wave 0 |
| ThemeProvider | src/components/theme.tsx | DNA token provider | Wave 0 |

### Wave 2+ (Builder Proposals)
| Component | Path | Usage | Proposed By | Wave |
|-----------|------|-------|-------------|------|
| GradientText | src/components/ui/gradient-text.tsx | DNA-colored gradient text | section-03-hero | Wave 2 |
| StaggerList | src/components/ui/stagger-list.tsx | Animated list with DNA timing | section-05-features | Wave 2 |

## Patterns Established
- Card pattern: [description, first used in section X]
- Stat counter: [description, first used in section Y]
```

Growth protocol ("builder proposes, orchestrator collects"):
1. Builder identifies reusable component during build
2. Builder lists it in SUMMARY.md under `reusable_components` (name, path, usage)
3. After wave completes, orchestrator reads all SUMMARY.md files for the wave
4. Orchestrator appends new reusable components to DESIGN-SYSTEM.md
5. Orchestrator includes "Shared Components Available" list in next wave's spawn prompts
6. No approval gate -- if a builder proposes it, it's listed. CD/QR can flag misuse during review

**Layer 3: Reviewer Feedback Loop**

Purpose: Quality learning -- patterns that worked, patterns that failed, accumulating over waves.
Location: Distributed (lives in CONTEXT.md feedback section + spawn prompt "Lessons Learned" section)
Size target: 10 lines in CONTEXT.md, 10 lines in spawn prompts
Update cadence: After every quality review cycle

Flow:
1. Quality reviewer completes wave review, produces GAP-FIX.md files + lessons learned summary
2. Orchestrator reads reviewer's lessons learned summary
3. Orchestrator writes top items to CONTEXT.md "Feedback Loop" section (max 10 lines)
4. Orchestrator embeds lessons in next wave's builder spawn prompts under "Lessons Learned" section
5. Format in spawn prompts:
   ```
   ### Lessons Learned (from previous waves)
   REPLICATE: [pattern scored well, which section, why]
   REPLICATE: [another pattern]
   AVOID: [pattern lost points, which section, how many points]
   AVOID: [another pattern]
   ```
6. Builders see this in their spawn prompt and adjust their approach accordingly

Platform memory integration:
- quality-reviewer.md has `memory: project` in frontmatter
- This enables the platform's built-in persistent memory for the reviewer
- Cross-session: reviewer remembers patterns from previous sessions automatically
- Memory is managed by Claude Code (200-line limit, auto-curated)
- This supplements (not replaces) the file-based feedback loop

**Cross-Layer Interactions:**
```
Builder SUMMARY.md
    |
    v
Orchestrator collects reusable_components
    |
    +---> DESIGN-SYSTEM.md (Layer 2: medium-term)
    |
    +---> Spawn prompt "Shared Components" (next wave builders see what's available)

Quality Reviewer lessons learned
    |
    v
Orchestrator aggregates
    |
    +---> CONTEXT.md "Feedback Loop" (Layer 1: short-term)
    |
    +---> Spawn prompt "Lessons Learned" (next wave builders learn from mistakes)
    |
    +---> Reviewer memory (Layer 3: cross-session, platform-managed)
```

**discussion-protocol.md** (~150-200 lines):

Update the v6.1.0 discussion-protocol.md for v2.0's pipeline architecture. The discussion protocol enforces human-in-the-loop verification at critical decision points. In v2.0, the protocol is referenced by multiple agents, not just the design-lead.

Read the existing v6.1.0 `agents/discussion-protocol.md` first for the current format and adapt it.

Key updates for v2.0:

**Who follows this protocol:**
- Build-orchestrator: before starting a wave (present wave plan + sections)
- Creative-director: when flagging a section as below bar (present issues + proposed improvements)
- Section-planner: when presenting master plan (layout assignments, beat sequence, wave structure)
- Any agent: when deviating from an approved plan

**When to invoke (decision gates):**
1. BEFORE first wave execution (present full master plan for approval)
2. BEFORE each wave (present wave plan: which sections, which builders, any concerns)
3. WHEN CD flags a section (present issues and required improvements for user awareness)
4. WHEN build fails (present failure + options: retry, skip, abort)
5. WHEN canary detects rot (present evidence + session boundary recommendation)
6. WHEN deviation from plan is needed (present reason + proposed change)

**Protocol steps:**
1. Present: What you're about to do (clear, specific, actionable)
2. Context: Why (reference the plan, DNA, creative vision)
3. Options: If applicable (usually "proceed" vs. "modify" vs. "skip")
4. Wait: For user response before proceeding
5. Log: Record decision in STATE.md or CONTEXT.md

**Format for presenting to user:**
```
## [Agent Name]: [Action Type]

### What
[1-2 sentences: what will happen]

### Why
[1-2 sentences: why this is the plan]

### Details
[Specifics: sections, files, dependencies]

### Options
1. **Proceed** -- [what happens if they approve]
2. **Modify** -- [what they can change]
3. **Skip** -- [what happens if they skip, if applicable]

Waiting for your input...
```

**When NOT to invoke:**
- auto tasks within an approved plan (builder executing PLAN.md tasks)
- CONTEXT.md rewrites (automated, no user input needed)
- Canary checks (automated, only notify if rot detected)
- SUMMARY.md writing (automated)
- Design system collection (automated)

**Error case:**
If an agent that should invoke the discussion protocol proceeds without waiting for user input, this is a protocol violation. The quality reviewer should flag this in review.
