**build-orchestrator.md** (~300-400 lines):
Frontmatter: name: build-orchestrator, description (coordinates wave-based execution, spawns parallel builders with pre-extracted context, manages session boundaries), tools: Read, Write, Edit, Bash, Grep, Glob, Task(section-builder, 3d-specialist, animation-specialist, content-specialist), model: inherit, maxTurns: 50.

System prompt must define:

**Input contract (STRICT -- orchestrator reads ONLY these):**
- CONTEXT.md (~80-100 lines) -- DNA identity, build state, creative notes, lessons learned, next instructions
- MASTER-PLAN.md -- wave map, section assignments, dependency graph
- Current wave's PLAN.md files (to extract context for spawn prompts)
- SUMMARY.md files from completed builders (to aggregate feedback + design system proposals)

The orchestrator does NOT read: DESIGN-DNA.md directly (gets DNA from CONTEXT.md), BRAINSTORM.md, CONTENT.md, REFERENCES.md, or any skill files. This prevents orchestrator overload (Pitfall 3 from research).

**Wave execution protocol:**
1. Read CONTEXT.md for current state and DNA identity
2. Read MASTER-PLAN.md for current wave sections
3. Verify all previous wave dependencies are COMPLETE
4. For each section in current wave: read its PLAN.md, read its builder_type field
5. Construct spawn prompts with Complete Build Context (template embedded in agent file)
6. Spawn parallel builders via Task tool (max 4 per wave)
7. Collect SUMMARY.md from each builder
8. Run post-wave coherence checkpoint
9. Aggregate lessons learned and design system proposals
10. Rewrite CONTEXT.md with updated state
11. Run canary check protocol
12. If 2+ waves completed in session: recommend session boundary

**Complete Build Context spawn prompt template** (~300 lines target):
Embed the full template from 02-RESEARCH.md "Pattern 2: Context Injection via Spawn Prompt". This is the exact prompt structure the orchestrator constructs for each builder. The spawn prompt includes the FULL Design DNA document (not a compressed extract) so builders have complete reference. Include all sections: Full Design DNA (~150 lines — complete DESIGN-DNA.md content including all 12 color tokens, type scale, spacing scale, fonts, signature element, motion language, forbidden patterns, tension plan, emotional arc template, and choreography defaults), Section Assignment (~15 lines), Beat Parameters (~8 lines), Adjacent Sections (~15 lines), Layout Patterns Used (~5 lines), Shared Components (~5 lines), Content for This Section (~30 lines), Quality Rules (~15 lines), Lessons Learned (~10 lines), YOUR TASK instruction.

**Post-wave coherence checkpoint:**
- Shadow consistency across sections
- Spacing consistency
- Background progression matches plan
- Layout diversity (no adjacent repeats)
- Typography hierarchy consistent
- Any issues found = blocking fix before next wave

**Builder type routing:**
Read `builder_type` from PLAN.md frontmatter. Spawn the corresponding agent: section-builder (default), 3d-specialist, animation-specialist, content-specialist.

**Session boundary management:**
- After every 2 waves: force-write CONTEXT.md, suggest new session
- Canary check after every wave (5 questions from memory, verify against CONTEXT.md)
- Turn 31+: mandatory session save

**Build failure handling:**
- Builder failure bubbles to user immediately
- Orchestrator does NOT auto-retry
- Failure pauses execution, asks user how to proceed
- Options: retry section, skip section, abort wave

**CONTEXT.md rewrite protocol:**
After each wave, rewrite (not append) CONTEXT.md with: DNA identity section (static), build state table (all sections with status/wave/layout/beat), creative direction notes (from CD), emotional arc position, feedback loop (lessons learned), next wave instructions.

**DESIGN-SYSTEM.md maintenance:**
After each wave, read builder SUMMARY.md files for `reusable_components` proposals. Append new components to `.planning/modulo/DESIGN-SYSTEM.md` with name, import path, usage note. Include "Shared Components Available" in subsequent spawn prompts from this file.
