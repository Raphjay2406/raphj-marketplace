**Layer 1: Decision Guidance (~100 lines)**

When to use: Whenever a failure occurs during build execution -- at any level (task, section, wave, session).

Severity classification decision tree -- the most critical content in this skill:

| Severity | Definition | Action | Who Decides |
|----------|-----------|--------|-------------|
| MINOR | Single task fails, no cascade risk. Examples: missing import, type mismatch, unused variable error | Auto-isolate and continue. Log to FAILURE-LOG.md. Report in next section completion highlight. | Claude auto-classifies |
| MAJOR | Task fails with potential cascade (other tasks depend on it) OR 2nd failure in same section. Examples: component won't render, API call fails, dependency missing | PAUSE wave. Present structured diagnosis with 2-3 fix options. Wait for user choice. | Claude diagnoses, user picks fix |
| CRITICAL | Wave-level failure OR 3+ failures in same wave OR context rot detected OR session interrupt. Examples: build won't compile at all, multiple sections failing, canary check 3+ failures | STOP the entire wave. Full diagnosis. User decides: fix + continue, restart wave, or start new session. | Claude diagnoses, user picks recovery path |

Escalation rules:
- 1st failure in a section: Classify per table above
- 2nd failure in same section: Escalate to MAJOR minimum
- 3rd failure in same section: Escalate to CRITICAL
- 3+ failures of same TYPE across sections: Systemic diagnosis (see Pattern 5)
- Canary check 2+ failures: CRITICAL -- context rot detected

When NOT to use (false alarms):
- Warning-only TypeScript output (no actual error) -- log as note, don't classify
- Intentional `@ts-expect-error` usage -- not a failure
- Test failure in a non-test context -- may be expected during development

Pipeline Connection:
- Referenced by: `design-lead` agent for wave-level error handling
- Referenced by: `build-orchestrator` agent for builder failure detection
- Referenced by: `section-builder` agent for task-level error handling and SUMMARY.md failure format
- Consumed at: Any point during build execution when an error occurs

**Layer 2: Award-Winning Examples (~200 lines)**

Pattern 1: **Structured Diagnosis Template**

Every MAJOR or CRITICAL failure produces this structured diagnosis:

```markdown
## Failure Report: [Section] / [Task]

**Type:** [build-error | type-error | runtime-error | dependency-error | timeout | context-rot]
**Severity:** [MINOR | MAJOR | CRITICAL]
**Location:** [file:line or agent context]

### What Happened
[1-2 sentences describing the exact error with code context]

### Root Cause
[Analysis of WHY this happened, not just WHAT happened]

### Fix Options
1. **[Quick fix name]** -- [description]
   - Trade-off: [what you give up]
   - Risk: [LOW | MEDIUM | HIGH]
   - Time: [estimate]
2. **[Proper fix name]** -- [description]
   - Trade-off: [what you give up]
   - Risk: [LOW | MEDIUM | HIGH]
   - Time: [estimate]
3. **[Alternative approach]** -- [description]
   - Trade-off: [what you give up]
   - Risk: [LOW | MEDIUM | HIGH]
   - Time: [estimate]

### Recommendation
Option [N] because [clear reasoning].

### Awaiting User Decision
Reply with option number (1, 2, or 3) to proceed.
```

Show a concrete example: TypeScript strict mode rejecting `animationTimeline` CSS property in a hero section. Show all 3 fix options with real trade-offs.

Pattern 2: **SUMMARY.md Failure Format (Section Builder Output)**

When a section builder fails, SUMMARY.md must include:

```markdown
## Section: [section-id]
**Status:** FAILED
**Wave:** [N]
**Completed Tasks:** [done]/[total]
**Failed At:** Task [N]: [task name]

### Failure Details
- **Type:** [error type]
- **File:** [file:line]
- **Error:** [exact error message]
- **Context:** [brief context for why this happened]

### Checkpoint State
- Task 1: COMPLETE ([brief description])
- Task 2: COMPLETE ([brief description])
- Task 3: FAILED ([what went wrong])
- Task 4: NOT_STARTED
- Task 5: NOT_STARTED

### Files Modified (Safe to Keep)
- [file path] (complete -- tasks 1-2)
- [file path] (partial -- task 3 in progress)

### Suggested Recovery
1. [Option 1]
2. [Option 2]
3. [Option 3]
```

Status values for SUMMARY.md: COMPLETE, FAILED, PARTIAL (some tasks done, builder stopped cleanly)

The design-lead/build-orchestrator MUST check SUMMARY.md after each builder finishes:
- SUMMARY.md exists + status COMPLETE -> success
- SUMMARY.md exists + status FAILED -> trigger diagnosis flow
- SUMMARY.md exists + status PARTIAL -> builder stopped at checkpoint, resume from there
- SUMMARY.md missing + files modified -> INTERRUPTED (crash/timeout), assess and recover
- SUMMARY.md missing + no files changed -> NOT_STARTED (builder never ran), restart

Pattern 3: **FAILURE-LOG.md Format**

Detailed failure history goes to `.planning/modulo/FAILURE-LOG.md` (not STATE.md):

```markdown
# Failure Log

## Entry [N] -- [timestamp]
**Section:** [section-id] | **Task:** [task-name]
**Type:** [error type] | **Severity:** [level]
**Error:** [exact error message]
**Root Cause:** [brief analysis]
**Resolution:** Option [N] -- [description]
**Duration:** [time to resolve]

---

## Entry [N-1] -- [timestamp]
...
```

FAILURE-LOG.md grows unbounded (append-only). STATE.md gets only the compact summary:

```markdown
## Recent Failures (5 most recent)
| Time | Section | Type | Severity | Resolution |
|------|---------|------|----------|------------|
| 14:28 | 02-nav | Build error | MINOR | Auto-fixed: missing import |
| 14:35 | 03-hero | Type error | MAJOR | User chose option 2: useScroll approach |
```

Pattern 4: **Checkpoint Resume Protocol**

For resuming after session interruption or crash:

Step 1: Pre-wave checkpoint (design-lead writes before spawning builders):
```markdown
## Checkpoint: Wave [N] Starting
**Sections:** [list of sections to build]
**Builders:** [count] parallel
**Started:** [timestamp]
```

Step 2: Detection after interruption (design-lead checks on resume):
- For each section in the interrupted wave:
  a. Check for SUMMARY.md -> Read status (COMPLETE, FAILED, PARTIAL)
  b. No SUMMARY.md + files modified in section directory -> INTERRUPTED
  c. No SUMMARY.md + no files -> NOT_STARTED

Step 3: Present resume options to user:
```markdown
### Session Resume: Wave [N] was interrupted

| Section | State | Recovery |
|---------|-------|----------|
| 03-hero | COMPLETE | Keep (5/5 tasks done) |
| 04-features | INTERRUPTED (task 3) | Resume from task 3 or restart |
| 05-proof | NOT_STARTED | Will build from scratch |

**Options:**
1. **Continue** -- keep completed work, resume interrupted sections from checkpoint, build unstarted sections
2. **Restart wave** -- discard all wave progress, rebuild from scratch
3. **New session** -- save state to CONTEXT.md, suggest starting fresh session

Recommendation: Option 1 (continue) -- saves ~12 minutes of completed work.
```

Step 4: For "continue" option:
- Completed sections: skip entirely
- Interrupted sections: read partial SUMMARY.md or assess files, resume from last completed task
- Not-started sections: build normally

Pattern 5: **Systemic Failure Escalation**

Track failure types across the build. If 3+ failures share the same type:

```markdown
### Systemic Issue Detected

**Pattern:** 3 type errors across sections 03-hero, 04-features, 06-cta
**Common cause:** All failures involve `animation-timeline` CSS property with TypeScript strict mode
**This is likely:** A project-level configuration issue, not individual section problems

**Recommended systemic fix:**
Add `animation-timeline` to the project's TypeScript CSS type declarations:
```typescript
// src/types/css.d.ts
declare module 'csstype' {
  interface Properties {
    animationTimeline?: string;
  }
}
```

**This will resolve:** All 3 current failures and prevent future occurrences.

**Awaiting User Decision:** Apply systemic fix? (yes/no)
```

Escalation threshold: 3+ failures of the same TYPE (not just same section).

Pattern 6: **Common Failure Types Reference Table**

| Type | Typical Cause | Typical Fix | Severity Default |
|------|---------------|-------------|-----------------|
| build-error | Missing import, syntax error | Add import, fix syntax | MINOR |
| type-error | TypeScript strict mode, wrong types | Type assertion, interface update | MINOR-MAJOR |
| runtime-error | Null reference, missing prop | Null check, default prop | MAJOR |
| dependency-error | Missing package, version conflict | Install package, resolve version | MAJOR |
| timeout | Builder took too long, infinite loop | Restart builder, check for loops | MAJOR |
| context-rot | Canary check failures, DNA drift | New session, re-read DNA | CRITICAL |
| style-drift | Output doesn't match DNA tokens | Re-read DESIGN-DNA.md, fix tokens | MAJOR |

**Layer 4: Anti-Patterns (~60 lines)**

Anti-Pattern 1: **Silent Retry** -- What: Automatically retrying a failed task without telling the user. Why wrong: User decision (Phase 2): "Build failures bubble to user (no auto-retry)." Instead: MINOR failures can be auto-FIXED (different from retry -- e.g., adding a missing import), but MAJOR/CRITICAL always present options.

Anti-Pattern 2: **Full Diagnosis in STATE.md** -- What: Writing 10-20 line failure reports directly in STATE.md. Why wrong: STATE.md has a 100-line budget. 3 failures would consume 30-60 lines. Instead: Compact 1-line summary in STATE.md, full details in FAILURE-LOG.md.

Anti-Pattern 3: **Ignoring Failed Builders** -- What: Orchestrator marks a section as complete without checking SUMMARY.md status. Why wrong: Silent failures lead to incomplete builds that appear complete. Instead: ALWAYS check SUMMARY.md status after each builder. Missing SUMMARY.md = potential failure.

Anti-Pattern 4: **Restarting Everything on Minor Failure** -- What: Killing the entire wave for a minor import error. Why wrong: Wastes completed work, destroys builder progress. Instead: Classify severity correctly. MINOR = auto-isolate. Only CRITICAL stops the wave.

Anti-Pattern 5: **No Checkpoint Before Wave** -- What: Spawning builders without writing the pre-wave checkpoint to STATE.md. Why wrong: If session crashes, no record of which builders were spawned, making recovery harder. Instead: Always write the pre-wave checkpoint before spawning any builders.

Anti-Pattern 6: **Treating Warnings as Failures** -- What: Classifying TypeScript warnings or ESLint warnings as failures. Why wrong: Warnings don't prevent building and don't need recovery. Instead: Log warnings as notes in SUMMARY.md, don't trigger the failure protocol.

**Machine-Readable Constraints:**

|-----------|----------|-------|-------------|

| MAJOR/CRITICAL user approval | Yes | Present options, wait for user | HARD -- no auto-retry |

| FAILURE-LOG.md for details | Yes | Full diagnosis in separate file | HARD -- STATE.md budget |
| STATE.md failure cap | Yes | Max 5 recent failures | HARD -- overflow to FAILURE-LOG.md |
