**Layer 1: Decision Guidance (~80 lines)**

When to use:
- After a complete build, user wants handoff artifacts for their design/development team
- User requests `/gen:export` or asks for "design system export"
- Team needs to maintain the design system in Storybook
- Other tools/platforms need design tokens (Figma sync-back, React Native, etc.)

When NOT to use:
- During the build itself -- export happens post-build only
- For a single-page site with no reusable components -- export may not provide value
- User only needs the built code -- the code IS the deliverable, export is optional

Decision tree for what to export (curation is Claude's discretion, per user decision):

**Export if:**
- Component is used in 2+ sections (reusable)
- Component has 3+ variants or states (complex enough to document)
- Component handles user interaction (needs testing documentation)
- Component is a layout primitive (SectionWrapper, Grid, Container)

**Skip export for:**
- One-off decorative elements (a specific section's unique background pattern)
- Simple text blocks with no interactivity
- Components that are just Tailwind utility compositions with no logic

Story depth decision tree:
- Visual-only story: Static component with no interactivity -> args + visual documentation
- Interaction story: Component with hover/click/keyboard -> add play functions
- Responsive story: Component with container query breakpoints -> viewport parameter story

Pipeline Connection:
- Referenced by: quality-reviewer during export completeness verification
- Consumed at: `/gen:export` command (post-build)
- Input from: Built components + DESIGN-DNA.md (token source)

Pattern 1: **Storybook 10 CSF Factories Setup**

Show the complete setup process:
1. Install: `npx storybook@latest init` (for Storybook 10)
2. `.storybook/preview.ts` configuration with DNA theme
3. `.storybook/main.ts` with story glob patterns

Then show the CSF Factories story format. CRITICAL: Use `preview.meta()` / `meta.story()` pattern, NOT the old `export default meta` / `satisfies Meta` pattern.

```typescript
// CORRECT: Storybook 10 CSF Factories
import preview from '../.storybook/preview';
import { Button } from './components/ui/Button';

const meta = preview.meta({
  component: Button,
  parameters: { layout: 'centered' },
  argTypes: {
    variant: { control: 'select', options: ['primary', 'secondary', 'ghost'] },
  },
});

export const Primary = meta.story({
  args: { variant: 'primary', children: 'Get Started' },
});
```

Show stories for 4 common component types:
- **Button** (with variant args, hover play function, keyboard play function)
- **Card** (with visual documentation, responsive viewport stories)
- **SectionWrapper** (with beat type parameter, DNA spacing demonstration)
- **Navigation** (with mobile/desktop stories, interaction play functions)

Each story must include at least one play function demonstrating interaction testing:

```typescript
import { userEvent, within, expect } from '@storybook/test';

export const HoverState = meta.story({
  args: { variant: 'primary', children: 'Hover Me' },
  play: async ({ canvasElement }) => {
    const canvas = within(canvasElement);
    const button = canvas.getByRole('button');
    await userEvent.hover(button);
    await expect(button).toHaveClass('hover');
  },
});

export const KeyboardNav = meta.story({
  args: { variant: 'primary', children: 'Tab to Me' },
  play: async ({ canvasElement }) => {
    const canvas = within(canvasElement);
    const button = canvas.getByRole('button');
    await userEvent.tab();
    await expect(button).toHaveFocus();
    await userEvent.keyboard('{Enter}');
  },
});
```

**Layer 3: Integration Context (~60 lines)**

DNA Connection:
- DESIGN-DNA.md is the source of truth for all token values
- Tailwind @theme variables are the runtime representation; DTCG tokens are the portable representation
- Both representations contain the same values, different formats

Archetype Variants:
- Export content varies by archetype (different color counts, different motion complexity)
- Some archetypes have more complex shadow systems (Glassmorphism, Neon Noir)
- Story count varies with component complexity per archetype

Pipeline Stage:
- Input from: Complete built project + DESIGN-DNA.md
- Output to: Storybook stories (for component documentation) + Token package (for multi-platform consumption)

Related Skills: design-dna (token source), tailwind-system (@theme as runtime tokens), design-system-scaffold (typed utilities that become exportable components)

**Layer 4: Anti-Patterns (~60 lines)**

Anti-Pattern 1: **CSF3 Instead of CSF Factories** -- What: Using `const meta = { component: X } satisfies Meta<typeof X>; export default meta;` pattern. Why wrong: Outdated Storybook 8 pattern. Storybook 10 uses CSF Factories. Instead: Use `preview.meta()` / `meta.story()` pattern.

Anti-Pattern 2: **Style Dictionary v3 Patterns** -- What: Using `require('style-dictionary')`, `value` instead of `$value`, CommonJS config. Why wrong: Style Dictionary 5 is ESM-only with DTCG format. Instead: ESM config, `$value`/`$type`/`$description` format.

Anti-Pattern 3: **Exporting Everything** -- What: Creating stories for every component including one-off decorative elements. Why wrong: Over-documentation creates noise, making the design system harder to navigate. Instead: Curate based on reusability, complexity, and interaction testing value.

Anti-Pattern 4: **Custom Token Format** -- What: Inventing a custom JSON schema for design tokens. Why wrong: W3C DTCG is the stable industry standard (2025.10). Custom formats require custom tooling and are not interoperable. Instead: Use DTCG format (`$value`, `$type`, `$description`).

Anti-Pattern 5: **Missing Play Functions** -- What: Stories that only show static renders of interactive components. Why wrong: Misses the key value of Storybook: proving interaction behavior works. Instead: Add play functions for hover, click, keyboard navigation on all interactive components.

Anti-Pattern 6: **Hardcoded Token Values** -- What: Putting literal values in token files instead of referencing DNA. Why wrong: Tokens drift from DNA if DNA is updated. Instead: Generate token files FROM DESIGN-DNA.md programmatically.

**Machine-Readable Constraints:**
