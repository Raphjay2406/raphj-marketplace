**Layer 1: Decision Guidance (~80 lines)**

When to use: Always. This skill is active during every build execution.

Reporting tier decision tree -- which tier fires when:

| Event | Tier | Format | Location |
|-------|------|--------|----------|
| Task starts | Tier 1 (task) | No report -- batched at completion | STATE.md only |
| Task completes | Tier 1 (task) | 1-line STATE.md update | STATE.md task table |
| Task fails | Tier 1 (task) | See error-recovery skill | STATE.md + FAILURE-LOG.md |
| Section complete | Tier 2 (section) | Compact inline highlight | Conversation |
| Wave complete | Tier 3 (wave) | Detailed summary template | Conversation + STATE.md |
| Build complete | Tier 4 (milestone) | Full report with scores | Conversation + STATE.md |
| User requests status | Any | Show current STATE.md task table | Conversation |

When NOT to use: Never -- reporting is always on during builds.

Pipeline Connection:
- Referenced by: `design-lead` agent during wave orchestration
- Referenced by: `build-orchestrator` agent during build execution
- Consumed by: `section-builder` agents for SUMMARY.md status format
- Consumed at: Every task completion, section completion, wave completion, build completion

**Layer 2: Award-Winning Examples (~200 lines)**

Pattern 1: **STATE.md Task Progress Table (Tier 1)**

Define the extended STATE.md format for per-task tracking. This table goes in STATE.md during active builds and is removed when build completes (to stay under 100 lines).

```markdown
## Task Progress (Wave 2 Active)
| Section | Task | Status | Started | Duration | Notes |
|---------|------|--------|---------|----------|-------|
| 03-hero | 1: Component shell | COMPLETE | 14:32 | 2m | - |
| 03-hero | 2: Typography | COMPLETE | 14:34 | 3m | - |
| 03-hero | 3: Animations | IN_PROGRESS | 14:37 | - | Scroll-driven parallax |
| 04-features | 1: Grid layout | COMPLETE | 14:33 | 4m | - |
| 04-features | 2: Cards | IN_PROGRESS | 14:37 | - | - |
| 05-proof | 1: Testimonials | PENDING | - | - | - |
```

Rules for this table:
- Update at task COMPLETION only (not task start) to reduce file I/O
- Batch multiple task completions in a single STATE.md write when possible
- Maximum 20 rows visible (current wave only). Previous wave tasks cleared.
- Status values: PENDING, IN_PROGRESS, COMPLETE, FAILED, SKIPPED
- Duration format: compact (2m, 5m, 12m)
- Notes: max 30 characters, key detail only

Pattern 2: **Section Completion Highlight (Tier 2)**

When a section completes (all tasks done + SUMMARY.md written), the design-lead reports a compact one-liner in conversation:

```
Section 03-hero complete (5/5 tasks, 8m) -- HOOK beat, full-bleed split layout
```

Format: `Section {id} complete ({tasks_done}/{tasks_total}, {duration}) -- {beat} beat, {layout_pattern} layout`

If a section had auto-fixed errors:
```
Section 04-features complete (5/5 tasks, 12m, 1 auto-fix) -- BUILD beat, asymmetric grid layout
```

If a section failed:
```
Section 05-proof FAILED at task 3/5 -- see error diagnosis below
```

Pattern 3: **Wave Summary Template (Tier 3)**

After all sections in a wave complete, the design-lead produces a detailed wave summary. This is the user's review gate -- builds pause here.

```markdown
### Wave 2 Complete

**Sections Built:** 03-hero, 04-features, 05-social-proof
**Duration:** 12 minutes
**Tasks:** 14/15 complete (1 auto-fixed minor error)

| Section | Tasks | Status | Layout Pattern | Beat |
|---------|-------|--------|---------------|------|
| 03-hero | 5/5 | COMPLETE | Full-bleed split | HOOK |
| 04-features | 5/5 | COMPLETE | Asymmetric grid | BUILD |
| 05-social-proof | 4/5 | COMPLETE (1 auto-fix) | Horizontal scroll | PROOF |

**Anti-Slop Score:** 28/35 (passing)
**DNA Compliance:** PASSED (all tokens match)
**Layout Diversity:** PASSED (3 distinct patterns)

**Canary Check:** 5/5 correct (context healthy)

**Next:** Wave 3 -- sections 06-cta, 07-footer
**Review Gate:** Awaiting user approval to proceed to Wave 3.
```

Rules for wave summary:
- Always include anti-slop score, DNA compliance, layout diversity
- Always include canary check results
- Always show next wave preview
- Always end with "Awaiting user approval" -- this is a hard gate

Pattern 4: **Milestone Report Template (Tier 4)**

After the final wave completes, produce the full build report:

```markdown
### Build Complete

**Total Duration:** 34 minutes across 4 waves
**Sections:** 8/8 complete
**Total Tasks:** 38/40 (2 auto-fixed minor errors)

**Quality Scores:**
- Anti-Slop Gate: 29/35 (PASSING, SOTD-competitive)

**Performance:**
- Lighthouse: 92/100
- FPS: All animations 60fps (no <30fps flags)

**Auto-Fixed Issues:** 2

**Screenshots:** Available at 4 breakpoints (375, 768, 1024, 1440px)
Run `/gen:verify` for full visual QA.
```

Pattern 5: **Screenshot Protocol**

Screenshots at 4 breakpoints (375, 768, 1024, 1440px):
- **After final wave:** Automatic. Playwright MCP captures all 4 breakpoints for each section. Saved to `.planning/genorah/screenshots/{section-name}-{width}px.png`
- **Mid-build:** Only on explicit request via `/gen:verify`. Not automatic to avoid performance overhead.
- **Wave summary:** Text-only summary (no screenshots). Reference screenshot availability for post-build review.

Pattern 6: **STATE.md Budget Management**

STATE.md has a 100-line budget (established in Phase 2). Progress reporting must stay within this:

Current wave task table: ~20 lines (current wave only, previous cleared)
Section status summary: ~15 lines (compact table, all sections)
Failure summary: ~5 lines (most recent 5, details in FAILURE-LOG.md)
Other state (phase, wave, config): ~20 lines (existing)
Total: ~60 lines active, leaving ~40 lines buffer

When build completes:
- Remove task progress table (no longer active)
- Keep section status summary (final state)
- Keep failure summary if any (for post-mortem)
- STATE.md drops back to ~40 lines

**Layer 3: Integration Context (~50 lines)**

DNA Connection:
- DNA compliance check compares built output against DESIGN-DNA.md token values
- Wave summary includes DNA compliance as a mandatory field
- Milestone report includes full DNA compliance audit

Archetype Variants:
- Reporting format is the same across all archetypes
- Anti-slop scores may vary (some archetypes are harder to score well on)
- No archetype-specific reporting differences

Pipeline Stage:
- Input from: Section builders (SUMMARY.md with task status), quality-reviewer (scores), live-testing (Lighthouse, FPS, axe-core)
- Output to: User (conversation highlights, wave summaries), STATE.md (machine-readable), CONTEXT.md (session continuity)

Related Skills: error-recovery (failure reporting format), quality-gate-protocol (scores consumed by reporting), live-testing (Lighthouse/FPS/accessibility data sources), anti-slop-gate (35-point scoring consumed by wave summary)

**Layer 4: Anti-Patterns (~50 lines)**

Anti-Pattern 1: **Verbose Per-Task Reporting** -- What: Multi-paragraph conversation reports for every task completion. Why wrong: Fills conversation with noise, leaving less context for building. Instead: Per-task is STATE.md only (machine-readable). Conversation gets per-section one-liners.

Anti-Pattern 2: **Mid-Build Screenshots** -- What: Automatic Playwright screenshots after every wave or section. Why wrong: Expensive (8+ screenshots per wave), slows the build, captures incomplete state. Instead: Automatic only after final wave. Mid-build on explicit request via /gen:verify.

Anti-Pattern 3: **Full Diagnosis in STATE.md** -- What: Putting complete error diagnosis (10-20 lines each) in STATE.md. Why wrong: Exceeds 100-line budget after 2-3 failures. Instead: Compact 1-line summary in STATE.md, full details in FAILURE-LOG.md.

Anti-Pattern 4: **Skipping Review Gates** -- What: Proceeding to next wave without user approval. Why wrong: User decision: review gate after every wave. Skipping removes user oversight. Instead: Always end wave summary with "Awaiting user approval" and wait.

Anti-Pattern 5: **Accumulating Previous Wave Data** -- What: Keeping all historical task progress in STATE.md. Why wrong: STATE.md grows linearly with build progress, exceeding budget. Instead: Current wave only in task table. Previous wave data in CONTEXT.md or FAILURE-LOG.md if needed.

**Machine-Readable Constraints:**
| Parameter | Required | Value | Enforcement |
|-----------|----------|-------|-------------|
| STATE.md budget | Yes | <= 100 lines | HARD -- trim older data if approaching |
| Wave review gate | Yes | User approval before next wave | HARD -- never auto-proceed |
| Screenshot breakpoints | Yes | 375, 768, 1024, 1440px | HARD -- all 4 required |
| Screenshot timing | Yes | After final wave only (auto) | SOFT -- mid-build on request |
| Task table scope | Yes | Current wave only | HARD -- clear previous wave tasks |
| Failure summary cap | Yes | Max 5 in STATE.md | HARD -- overflow to FAILURE-LOG.md |
