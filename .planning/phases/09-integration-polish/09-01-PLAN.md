**Layer 1: Decision Guidance (~100 lines)**

When to use:
- User has a Figma design they want to implement
- `/modulo:start-design --figma` is invoked
- Figma is the design intent source for visual QA comparison
- Code Connect components exist and should be reused

When NOT to use:
- Building from scratch (no Figma file) -- use normal start-design flow
- Figma file is a rough wireframe with no visual styling -- treat as inspiration, not import source

Decision tree for MCP tool selection (this is critical -- teach Claude which tool to use when):

| Situation | Tool | Why |
|-----------|------|-----|
| First look at a Figma file | `get_metadata` | Get sparse XML overview without exceeding token limits |
| Understanding a specific section | `get_design_context` on node ID | Full layout, styles, components for that node |
| Large frame returns truncated | Break into child nodes, `get_design_context` per child | Avoid token limit truncation |
| Need visual reference for QA | `get_screenshot` | Captures the design as-rendered |
| Extracting design tokens | `get_variable_defs` | Returns Figma variables (colors, spacing, typography) |
| Checking for mapped components | `get_code_connect_map` | Returns components already linked to code |
| Finding potential mappings | `get_code_connect_suggestions` | Auto-detected mapping opportunities |
| Creating new mappings | `add_code_connect_map` | Links Figma components to codebase components |
| Pushing code as design | `generate_figma_design` | Code-to-Figma direction (rare, for sync-back) |

Pipeline Connection:
- Referenced by: `figma-translator` agent during import workflow
- Referenced by: `quality-reviewer` agent during visual QA verify step
- Consumed at: `/modulo:start-design --figma` and `/modulo:verify` visual QA

Pattern 2: **Hybrid DNA-Figma Token Resolution Protocol**

1. Figma Variable (via `get_variable_defs`) -> Use directly

3. Raw hex (not in Figma variables) -> FLAG for user decision with 3 options: (a) map to existing DNA token, (b) add as new DNA token, (c) keep as-is with documented deviation
4. Not in Figma at all -> DNA provides (motion, interactions, forbidden patterns)

Show a concrete example: a Figma component with #FF6F3C as accent, mapping it to DNA accent-1 token. Show the flag format for unmapped colors.

**Layer 3: Integration Context (~80 lines)**

DNA Connection:
- ALL 12 DNA color tokens are the mapping target for Figma colors
- Figma typography maps to DNA type scale (8 levels)
- Figma spacing maps to DNA spacing scale (5 levels)
- Motion, forbidden patterns, signature element come from DNA (not Figma)
- Hybrid relationship: Figma where present, DNA fills gaps

Archetype Variants:
- Import does NOT change the archetype -- the archetype was set during start-design
- Figma may introduce visual elements that conflict with archetype forbidden patterns -- FLAG these
- Creative tension and wow moments still come from the archetype, not Figma

Pipeline Stage:
- Input from: Figma file URL provided by user + DESIGN-DNA.md from start-design phase
- Output to: PLAN.md files consumed by /modulo:execute + Figma screenshots for visual QA

Related Skills: design-dna (token mapping target), emotional-arc (beat assignment for sections), quality-gate-protocol (visual QA during verify), live-testing (Playwright screenshots)

**Layer 4: Anti-Patterns (~80 lines)**

Anti-Pattern 1: **Direct Code Generation from Figma** -- What: Generating React/TSX code directly from Figma data instead of producing PLAN.md files. Why wrong: Bypasses user review, skips quality enforcement, misses beat assignment and emotional arc. Instead: Always produce PLAN.md files that flow through the normal execute pipeline.

Anti-Pattern 2: **get_design_context on Full Pages** -- What: Calling get_design_context on an entire page node. Why wrong: Token limit truncation causes missing components and styles. Instead: Call get_metadata first, then get_design_context on individual section nodes.

Anti-Pattern 3: **Hardcoding Figma Colors Over DNA Tokens** -- What: Using raw hex values from Figma instead of mapping to DNA tokens. Why wrong: Breaks theming, dark mode, and DNA enforcement. Instead: Map to DNA tokens using the hybrid resolution protocol. Flag unmapped colors for user decision.

Anti-Pattern 4: **Trusting get_variable_defs Aliases** -- What: Assuming get_variable_defs returns alias chains (semantic -> primitive references). Why wrong: Known limitation -- it returns resolved values, not alias structure. Instead: Cross-reference with DNA token system to reconstruct semantic relationships. Map by value matching.

Anti-Pattern 5: **Ignoring Code Connect Mappings** -- What: Generating new components for everything instead of checking for existing Code Connect mappings. Why wrong: Wastes effort, creates duplicate components, misses project-specific implementations. Instead: Always call get_code_connect_map first and reuse mapped components.

Anti-Pattern 6: **Visual QA Without Matching Viewports** -- What: Comparing Figma screenshot at one viewport size with Playwright screenshot at another. Why wrong: Every pixel difference is noise, not signal. Instead: Match viewport dimensions exactly between Figma screenshot and Playwright screenshot.

**Machine-Readable Constraints:**
| Parameter | Required | Value | Enforcement |
|-----------|----------|-------|-------------|
| get_metadata before get_design_context | Yes | Always call metadata first | HARD -- prevents truncation |
| Non-token color flagging | Yes | All raw hex values flagged | HARD -- user must decide |
| PLAN.md output (not direct code) | Yes | Always produce PLAN.md | HARD -- pipeline requirement |
| Visual QA viewport match | Yes | Same dimensions both screenshots | HARD -- diff is meaningless otherwise |
| Code Connect check | Yes | get_code_connect_map before generating | SOFT -- should check but can skip if clearly no mappings |
