**Layer 1: Decision Guidance must include:**

1. **When to Use** -- Any Genorah project that connects to external services. Specific triggers:
   - Contact/lead forms that submit to a CRM (HubSpot, Salesforce) or email notification
   - Receiving incoming webhooks from external services (Stripe, GitHub, etc.)
   - Calling any external API that requires authentication (API keys, OAuth tokens)
   - Sending transactional emails (confirmations, notifications)
   - Any form needing spam protection (Cloudflare Turnstile)
   - Agent needs current API documentation during build (Context7 MCP)

2. **When NOT to Use** -- Redirect to other skills when this one is the wrong choice:
   - **GraphQL APIs** -- Deferred to v2 (API-V2-01). This skill covers REST/HTTP APIs only
   - **Real-time WebSocket/SSE** -- Deferred to v2 (API-V2-02)
   - **Client-side caching (SWR, TanStack Query)** -- Deferred to v2 (API-V2-03). This skill covers server-side patterns only
   - **CMS content fetching** -- Use existing `cms-integration` skill. This skill covers API endpoints for form submissions and webhooks, not CMS data fetching
   - **Authentication flows (login, signup, sessions)** -- Use `auth-ui` skill. This skill covers server-side API proxying, not auth UX
   - **SSR/ISR/streaming rendering** -- Use the upcoming `ssr-dynamic-content` skill (Phase 19)

3. **Integration Type Decision Tree** -- Key branching decisions:
   - **Form collects leads/contacts?**
     - Yes, using HubSpot -> Use HubSpot Forms API v3 pattern
     - Yes, using Salesforce (simple marketing) -> Use Web-to-Lead pattern
     - Yes, using Salesforce (programmatic/complex) -> Use REST API with JWT OAuth pattern
     - Yes, generic CRM/service -> Use generic webhook POST pattern
     - No CRM, just email notification -> Use Resend or SendGrid pattern
   - **Receiving external events?**
     - From Stripe -> Use Stripe webhook pattern (SDK-based verification)
     - From GitHub -> Use GitHub webhook pattern (HMAC-SHA256)
     - From other provider -> Use generic HMAC-SHA256 verifier pattern
   - **Calling external API from server?**
     - Simple, one-off call -> Use thin fetch wrapper pattern
     - Multiple calls, production use -> Use typed API client pattern (with retry + rate limiting)
   - **Need spam protection on forms?**
     - Yes -> Add Cloudflare Turnstile (privacy-friendly, no visual puzzle)
   - **Need current API docs during build?**
     - Yes -> Use Context7 MCP with fallback chain

4. **Server-Side Proxy Principle** -- Brief explanation (3-5 sentences) of WHY all external API calls go through server-side code. Core rule: secrets never reach the client bundle. Framework-specific prefix rules:
   - Next.js: `NEXT_PUBLIC_` prefix exposes to client. Secrets MUST NOT use this prefix.
   - Astro: `PUBLIC_` prefix exposes to client. Secrets MUST NOT use this prefix.
   - Vite: `VITE_` prefix exposes to client. Secrets MUST NOT use this prefix.
   - **Genorah convention:** ALL secrets use unprefixed env var names. Only truly public values (like Turnstile site key) use the framework public prefix.

5. **Context7 MCP Integration** -- Subsection explaining:
   - What Context7 is: MCP server providing two tools (`resolve-library-id` and `query-docs`)
   - Which agents have access: design-researcher (primary), specialist agents (for complex APIs), quality-reviewer (for verification). Section-builder does NOT use Context7 (latency concern)
   - Explicit triggers for lookup: version-specific API question, unfamiliar SDK, deprecated pattern detected, rate limit question
   - Curated library list: Table with library names and recommended query topics (resend, stripe, @sendgrid/mail, next, astro, @hubspot/api-client, zod)
   - Fallback chain: (1) Context7 MCP -> (2) Hardcoded skill baseline -> (3) Official docs URL -> (4) Flag to user
   - Important: Context7 is a freshness upgrade tool, NOT a runtime dependency. The skill contains hardcoded, version-pinned baseline patterns that work without Context7.

6. **Pipeline Connection:**
   - Referenced by: section-builder (when generating forms, API integrations), specialist agents (when integrating external services mid-build)
   - Context7 access: design-researcher (Track 3 Component Library), specialist agents, quality-reviewer
   - Consumed at: `/gen:execute` Wave 2+ for form sections, API integration sections
   - Input from: `/gen:start-project` (project requirements identify needed integrations), `/gen:plan-dev` (sections flagged for API integration)
   - Output to: Server actions, API route handlers, API endpoints, utility libraries, .env.example

7. **Timeout Defaults** -- Small table documenting the opinionated timeout defaults from CONTEXT decisions:
   | Category | Timeout | Retries | Rationale |
   |---|---|---|---|
   | Form submissions | 5,000ms | 3 (1s/2s/4s) | User is waiting; fast fail preferred |
   | CRM API calls | 10,000ms | 3 (1s/2s/4s) | CRM APIs can be slow; reasonable patience |
   | File uploads | 30,000ms | 1 (no retry) | Large payloads; retry is wasteful |
   | Webhook receiving | No timeout | 0 (no retry) | Providers retry on their own; respond quickly |
