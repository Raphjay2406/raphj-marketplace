**Layer 1: Decision Guidance (When/Why)**
- WHY progressive enforcement: Problems caught at Layer 1 (build-time) cost nothing to fix. Problems caught at Layer 4 (user checkpoint) cost the most. Progressive enforcement catches problems at the cheapest fix point.
- WHY tiered severity (LOCKED DECISION): Not all findings are equal. Critical failures (anti-slop <25, Lighthouse <80, forbidden patterns) must block the pipeline. Warnings (minor spacing, close-to-threshold scores) should accumulate but not block. This prevents both: (a) shipping with critical issues, and (b) alert fatigue from treating everything as critical.
- WHY conditional user checkpoints (LOCKED DECISION): Clean builds auto-proceed without user intervention. Only builds with accumulated warnings require user review. This respects the user's time while ensuring they see genuine concerns.
- WHY CD + QR in parallel (LOCKED DECISION): Creative quality (CD) and technical quality (QR) are orthogonal concerns. Running them sequentially adds latency. Running them in parallel with merged findings is faster with no quality loss.
- Build failures escalate to user (LOCKED DECISION from Phase 2): No auto-retry, no autonomous fix attempts. The system reports and waits for user direction.

**Layer 2: Award-Winning Examples**

**The 4-Layer Progressive Enforcement System:**

Define each layer with: WHEN it fires, WHO runs it, WHAT it checks, HOW findings are handled.

```
LAYER 1: BUILD-TIME (cheapest to fix)
├── When: During each section build, as part of builder's final tasks
├── Who: Section-builder agent (self-check)
├── What:
│   ├── must_haves.artifacts: All files exist and non-empty
│   ├── must_haves.truths: All assertions hold (responsive, animations, states)
│   ├── DNA compliance: No hardcoded colors, fonts, spacing
│   ├── Layout pattern: Built section matches assigned pattern from PLAN.md
│   └── Light polish: Basic hover states, focus management, reduced-motion
├── Failure action:
│   ├── Builder retries the failing task (within same build, max 2 retries)
│   └── If still failing after retries: mark section as INCOMPLETE in SUMMARY.md
├── Cost: Free (embedded in builder's execution)
└── Outputs: Section SUMMARY.md with self-check results

LAYER 2: POST-WAVE (catch before next wave starts)
├── When: After ALL sections in a wave complete
├── Who: Quality-reviewer + Creative-director (in PARALLEL)
├── What (Quality Reviewer):
│   ├── 3-level goal-backward verification (Existence, Substantive, Wired)
│   ├── Anti-slop 35-point scoring per section
│   ├── Cross-section consistency check (DNA token usage, spacing rhythm)
│   └── Layout diversity validation (does built output match MASTER-PLAN.md assignment?)
├── What (Creative Director):
│   ├── Archetype personality check (does section feel like the archetype?)
│   ├── Creative tension assessment (bold enough? PEAK sections must have tension)
│   ├── Emotional arc accuracy (does section hit its assigned beat?)
│   ├── Color journey assessment (colors evolve across sections, not static)
│   └── Screenshot-worthy? (would this section be shared?)
├── Findings merge:
│   ├── Both agents write findings independently
│   ├── Build-orchestrator merges into single quality report
│   └── Each finding classified by severity (CRITICAL / WARNING / INFO)
├── Failure action:
│   ├── CRITICAL: Pipeline blocks immediately. Escalate to user with evidence
│   ├── WARNING: Add to running tally. Continue to next wave
│   └── INFO: Log in report, no action needed
├── Gap fixes: QR/CD create GAP-FIX.md files. Polisher processes them before next wave
├── Cost: One review cycle per wave (both agents run in parallel, so ~1 cycle, not 2)
└── Outputs: Quality report, GAP-FIX.md files (if issues), updated running tally

LAYER 3: END-OF-BUILD (comprehensive final check)
├── When: After all waves complete AND polish pass complete
├── Who: Quality-reviewer with MCP tools (live-testing skill)
├── What:
│   ├── 4-breakpoint responsive screenshots (375, 768, 1024, 1440px)
│   ├── Lighthouse performance audit (hard fail < 80)
│   ├── axe-core accessibility audit (hard fail on critical violations)
│   ├── Animation FPS monitoring (hard fail < 30fps)
│   ├── Full-page anti-slop scoring (holistic, not per-section)
│   ├── Awwwards 4-axis scoring (Design, Usability, Creativity, Content -- each /10)
│   └── Screenshot comparison against PLAN.md reference targets
├── Failure action:
│   ├── Lighthouse < 80: CRITICAL -- escalate with specific bottlenecks
│   ├── Critical a11y violation: CRITICAL -- escalate with violation details
│   ├── FPS < 30: CRITICAL -- escalate with section/animation identified
│   ├── Visual regression: WARNING -- flag with screenshot comparison
│   └── Low Awwwards prediction: WARNING -- log with dimension breakdown
├── Cost: Full browser testing cycle (most expensive layer)
└── Outputs: Live Testing Report, updated running tally, Awwwards prediction

LAYER 4: USER CHECKPOINT (human judgment)
├── When: After Layer 3 completes
├── Who: Build-orchestrator presents, user decides
├── What: User reviews:
│   ├── 4-breakpoint screenshots
│   ├── Quality report (anti-slop score, Awwwards prediction)
│   ├── CD creative assessment
│   ├── Warning tally summary
│   └── Critical issues (if any -- should have been caught at earlier layers)
├── Trigger logic:
│   ├── Warnings accumulated → MANDATORY checkpoint
│   │   "Build complete with [N] warnings. Review required before shipping."
│   │   Present: screenshot gallery, warning list, quality scores, CD assessment
│   │   User options: "ship" / "iterate on [specific issues]" / "fix warnings"
│   └── Clean build (no warnings, no criticals) → AUTO-PROCEED
│       "Clean build verified. Awwwards prediction: [X.X]. Ready to ship."
│       Log result, continue without blocking
├── Cost: Human time (minimized by conditional triggering)
└── Outputs: User decision (ship / iterate / fix)
```

**Severity Classification System:**

Define every condition that triggers each severity level:

```markdown
## CRITICAL (blocks pipeline, escalates to user immediately)
- Anti-slop score < 25/35
- Archetype forbidden pattern detected in built code
- Missing Design DNA signature element entirely
- Lighthouse performance score < 80
- Critical accessibility violation (axe-core impact: "critical")
- Animation FPS < 30 on any section's scroll animation
- Build-time failure (missing artifact, broken import, component won't render)
- Section built with wrong layout pattern (doesn't match MASTER-PLAN.md assignment)

## WARNING (running tally, accumulates across waves)
- Anti-slop score 25-27 (passing but low, room for improvement)
- Minor spacing inconsistencies (e.g., inconsistent padding between similar sections)
- Non-critical accessibility suggestions (axe-core impact: "serious" or "moderate")
- Close-to-threshold Lighthouse scores (80-85 performance)
- Average FPS 30-45 (technically passing but concerning)
- CD flags section as "could be bolder" (below creative bar but not failing)
- Missing non-critical polish items from universal checklist
- Visual regression from PLAN.md expectations (1 attribute BELOW)
- Background token not alternating between adjacent sections

## INFO (logged in report, not tallied)
- Anti-slop score breakdown per category (informational)
- Awwwards 4-axis prediction scores (informational)
- Performance metrics above thresholds (good to know)
- Screenshot comparison notes (no issues, just documentation)
- Minor axe-core suggestions (impact: "minor")
- Lessons learned patterns (for future waves)
- FPS readings above threshold (just recording)
```

**Running Tally Format:**

Define exactly how the build-orchestrator tracks and displays warnings:

```markdown
## Build Quality Status

### Overall: IN_PROGRESS | Wave [N] of [M]
### Critical Issues: [N] ([blocking / none])
### Warning Tally: [N] accumulated

| # | Wave | Section | Warning | Source | Severity |
|---|------|---------|---------|--------|----------|
| 1 | 1 | 02-logos | Lighthouse performance 83 (threshold: 80) | QR | warning |
| 2 | 2 | 04-product | Missing alt text on 1 image (serious) | QR/axe | warning |
| 3 | 2 | 05-stats | Creative tension could be bolder | CD | warning |
| 4 | 3 | 07-testimonials | Spacing inconsistency in card grid | QR | warning |

### Anti-Slop Scores by Section
| Section | Score | Rating |
|---------|-------|--------|
| 01-hero | 30/35 | SOTD-READY |
| 02-logos | 28/35 | STRONG |
| 03-features | 27/35 | PASS |
| ... | ... | ... |
| Average | 28.4/35 | STRONG |

### Health: GOOD | CONCERNING | CRITICAL
- GOOD: No critical issues, warnings < 5
- CONCERNING: No critical issues, warnings 5-10
- CRITICAL: Any critical issue present (should have blocked already)
```

This status block is maintained in STATE.md (or a dedicated QUALITY-STATUS.md) and updated after every wave.

**Layer 3: Integration Context**
- Build-orchestrator is the primary consumer: it implements the timing (when each layer fires) and manages the running tally
- Quality-reviewer uses severity classification when writing GAP-FIX.md and the Live Testing Report
- Creative-director uses severity classification for its creative findings (below-bar = WARNING, forbidden pattern = CRITICAL)
- Polisher is invoked between waves when GAP-FIX.md files exist, and at end-of-build for the full polish pass
- This skill ORCHESTRATES the other Phase 4 skills:
  - reference-benchmarking: Comparison happens at Layer 2 (post-wave) and Layer 3 (screenshot comparison)
  - compositional-diversity: Validated at Layer 1 (builder self-check) and Layer 2 (QR validates)
  - polish-pass: Polisher runs between Layer 2 and next wave (gap fixes) and before Layer 3 (full polish)
  - live-testing: Executed at Layer 3 (end-of-build comprehensive testing)

Define the complete wave execution timeline showing exactly when each Phase 4 skill activates:
```
Pre-Build:
  section-planner → uses reference-benchmarking + compositional-diversity skills

Wave N:
  [Layer 1] Builders execute with assigned patterns and reference targets

  [Layer 2] GAP-FIX.md created for issues
```

**Layer 4: Anti-Patterns**
- Running Layer 3 tests per wave (comprehensive browser testing should run ONCE at end, not per wave -- that's Layer 2's job)
- Auto-retrying on critical failures (LOCKED DECISION: failures escalate to user, no autonomous retry)
- Treating all findings as equal severity (the whole point of tiered severity is proportionate response)
- Warning fatigue from too many INFO items in the tally (only CRITICAL and WARNING items go in the tally -- INFO is logged but not tallied)
- Skipping Layer 4 because there are no criticals (the checkpoint is conditional on WARNINGS, not just criticals)
- Running CD and QR sequentially (they're independent and should run in parallel to save time)
- Blocking on warnings (warnings accumulate but don't block -- only criticals block)
- User checkpoint on every wave (checkpoint is at Layer 4, end-of-build only. Per-wave, only criticals escalate)
