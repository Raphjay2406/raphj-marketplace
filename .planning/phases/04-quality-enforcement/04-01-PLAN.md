**Layer 1: Decision Guidance (When/Why)**
- WHY reference targets: Builders produce higher quality when they have a concrete visual bar, not abstract instructions. "Match Linear's hero" is more actionable than "make it award-winning."
- WHICH sections get reference targets: Key beats only -- HOOK (Hero), PEAK, CLOSE, and any section with creative tension level 3+. Supporting beats (TEASE, BUILD, BREATHE, PROOF) rely on DNA + archetype constraints alone.
- Include a decision table:
  | Beat Type | Gets Reference Target? | Reason |
  |-----------|----------------------|--------|
  | HOOK (Hero) | YES | First impression, highest impact |
  | PEAK | YES | Screenshot moment, maximum wow |
  | CLOSE (CTA) | YES | Conversion point, must be compelling |
  | TENSION (level 3+) | YES | Creative tension needs a quality bar |
  | TEASE | No | Low complexity, DNA sufficient |
  | BUILD | No | Dense/functional, organization is quality |
  | BREATHE | No | Minimal elements, restraint is quality |
  | PROOF | No | Content-driven, specificity is quality |
- HYBRID approach (LOCKED DECISION): Curated library provides per-archetype baseline references. Per-project research by the researcher agent supplements with industry-specific current winners for key sections.
- Reference targets go in each section's PLAN.md so builders see them before building.

**Layer 2: Award-Winning Examples**
Define the reference_target PLAN.md frontmatter format:
```yaml
reference_target:
  url: "https://example-award-winner.com"
  section: "hero"
  quality_bar: "Split layout with product screenshot, gradient text, staggered reveal"
  screenshot: ".planning/modulo/references/XX-section-target.png"
```

Define the `<reference_quality_target>` block format for PLAN.md body:
```markdown
<reference_quality_target>
**Reference:** [Site Name] - [Section Type]
**Source:** [URL or screenshot path]
**Beat:** [Beat type]

**Quality Attributes to Match or Exceed:**
1. Layout: [specific composition, ratios, alignment, overlap]
2. Typography: [font effects, tracking, gradient, weight contrast]
3. Color: [layer count, shadow colors, gradient complexity]
4. Motion: [entrance choreography, scroll integration, timing]
5. Depth: [shadow layers, glass effects, overlaps, perspective]
6. Micro-detail: [textures, borders, cursor effects, selection color]

**Quality Bar Question:** Would this section win a direct visual comparison against the reference?
</reference_quality_target>
```

Each attribute must be SPECIFIC -- not "nice typography" but "display font at -0.04em tracking, gradient from white to white/40". Show 3 complete examples:
1. A Neo-Corporate hero reference target (Linear/Vercel style)
2. A Kinetic peak section reference target (heavy animation, scroll-driven)
3. An Ethereal CTA/close section reference target (soft, spacious, emotional)

**Curated Reference Library Structure:**
Define the per-archetype library format. Start with the top 5 archetypes (Neo-Corporate, Kinetic, Ethereal, Editorial, Luxury) with 3-5 reference entries each. Other archetypes get archetype-level quality attributes without specific site references. Structure:
```markdown
## Archetype: [Name]
### Quality Personality
[3-5 sentences defining what "excellent" means for this archetype]

### Reference Sites
| Site | Archetype Fit | Key Sections | Date Added |
|------|--------------|-------------|------------|
| [name/url] | [why it exemplifies this archetype] | Hero, Features, CTA | [date] |

### Per-Section Quality Attributes
#### Hero (HOOK)
- Layout: [what great heroes look like for this archetype]
- Typography: [specific type treatments]
- Motion: [entrance choreography style]
- Signature: [what makes it distinctly this archetype]
```

Include guidance for the researcher agent on supplementing the curated library with per-project industry-specific references during the research phase. The researcher looks for current Awwwards/SOTD winners in the target industry and extracts the same 6 quality attributes.

**Layer 3: Integration Context**
- Section-planner reads this skill to generate reference_target blocks in PLAN.md
- Quality-reviewer reads this skill to know HOW to compare output against targets (using Claude's multimodal capability to read screenshots and compare against text specifications)
- Researcher agent uses the curated library as starting point and supplements with per-project research
- Reference comparison is NOT pixel-perfect matching -- it's "would this win a direct comparison?" judgment by the quality-reviewer

Define the quality-reviewer comparison protocol:
1. Read the section's `<reference_quality_target>` from PLAN.md
2. Read/view the built section's code (or screenshot if available)
3. For each of the 6 quality attributes: assess whether the built section matches or exceeds the reference
4. Write a comparison verdict: EXCEEDS / MATCHES / BELOW for each attribute
5. If 2+ attributes are BELOW: flag as a gap in GAP-FIX.md
6. If overall verdict is BELOW: this is a WARNING (not critical, since reference targets are aspirational)

**Layer 4: Anti-Patterns**
- Reference targets that are just URLs without extracted quality attributes (builder gets no actionable guidance)
- Reference targets on every section (over-constraining supporting sections that should breathe)
- Stale references from 2+ years ago that look dated
- Reference targets that conflict with the project's archetype (e.g., using a Brutalist reference for an Ethereal build)
- Pixel-perfect matching expectations (the reference sets a quality BAR, not a design to copy)
- Using competitor sites as references without extracting the quality attributes (builder might copy instead of compete)
