**Layer 1: Decision Guidance (When/Why)**
- WHY live testing: Code review catches structural issues but misses visual and performance problems. A section can pass anti-slop scoring but have poor Lighthouse performance or invisible accessibility violations.
- WHEN it runs (LOCKED DECISION): Once at end of build, AFTER the polish pass completes. Tests the final polished output. Never per-section, never per-wave (that kills velocity).
- WHAT blocks (LOCKED DECISIONS from CONTEXT.md):
  - Lighthouse performance score < 80 = CRITICAL FAIL (hard fail, blocks pipeline)
  - Critical accessibility violation (axe-core impact: "critical") = CRITICAL FAIL
  - Animation FPS < 30 on any section = CRITICAL FAIL
  - Lighthouse accessibility < 90 = WARNING
  - Lighthouse best practices < 80 = WARNING
  - axe-core "serious" impact = WARNING
  - Average FPS 30-45 = WARNING
- CD review and automated testing run in PARALLEL (LOCKED DECISION) -- both produce findings merged into a single quality report.
- TOOL STRATEGY: Primary tool is Playwright MCP (`@playwright/mcp@latest`). If unavailable, graceful degradation path defined.

**Preconditions before running this protocol:**
1. All waves complete (every section built)
2. Polish pass complete (polisher has finished)
3. Dev server running at localhost:[port]
4. Playwright MCP server available (check and handle gracefully if not)

**Step 2: Lighthouse Performance Audit**

Primary approach (CLI via Bash tool):
```bash
npx lighthouse http://localhost:[port] \
  --output=json \
  --output-path=.planning/modulo/audit/lighthouse.json \
  --chrome-flags="--headless --no-sandbox" \
  --only-categories=performance,accessibility,best-practices
```

Parse results and classify:
```
Performance score:
  < 80 → CRITICAL FAIL (pipeline blocks, escalate to user)
  80-89 → WARNING (close to threshold)
  90+ → PASS

Accessibility score:
  < 90 → WARNING
  90+ → PASS

Best Practices score:
  < 80 → WARNING
  80+ → PASS
```

Key performance metrics to extract and report:
- LCP (Largest Contentful Paint): target < 2.5s
- CLS (Cumulative Layout Shift): target < 0.1
- TBT (Total Blocking Time): target < 200ms
- FCP (First Contentful Paint): target < 1.8s
- Speed Index: target < 3.4s

If Lighthouse CLI is not available: note in report, recommend user install (`npm install -g lighthouse`), proceed with remaining tests.

**Step 3: axe-core Accessibility Audit**

Primary approach (CDN injection via Playwright_evaluate):
```javascript
// Inject axe-core from CDN
await Playwright_evaluate({
  code: `
    return new Promise((resolve, reject) => {
      const script = document.createElement('script');
      script.src = 'https://cdnjs.cloudflare.com/ajax/libs/axe-core/4.9.1/axe.min.js';
      script.onload = () => {
        axe.run(document, {
          runOnly: { type: 'tag', values: ['wcag2a', 'wcag2aa', 'wcag21a', 'wcag21aa'] }
        }).then(results => {
          resolve({
            violations: results.violations.map(v => ({
              id: v.id,
              impact: v.impact,
              description: v.description,
              nodes: v.nodes.length,
              help: v.helpUrl
            })),
            passes: results.passes.length,
            incomplete: results.incomplete.length
          });
        });
      };
      script.onerror = () => reject('Failed to load axe-core from CDN');
      document.head.appendChild(script);
    });
  `
});
```

Classify violations by impact:
```
impact: "critical" → CRITICAL FAIL (pipeline blocks)
impact: "serious"  → WARNING (running tally)
impact: "moderate" → INFO (logged in report)
impact: "minor"    → INFO (logged in report)
```

Fallback if CDN injection fails: instruct user to `npm install @axe-core/playwright` in target project. Note the fallback in the report.

**Step 4: Animation FPS Monitoring**

Inject FPS counter and scroll through page:
```javascript
// FPS monitoring script -- inject via Playwright_evaluate
(function monitorFPS() {
  return new Promise((resolve) => {
    const readings = [];
    let frames = 0;
    let lastTime = performance.now();
    const duration = 3000; // Monitor for 3 seconds
    const startTime = performance.now();

    function tick(now) {
      frames++;
      const elapsed = now - lastTime;
      if (elapsed >= 500) { // Sample every 500ms
        readings.push(Math.round(frames / (elapsed / 1000)));
        frames = 0;
        lastTime = now;
      }
      if (now - startTime < duration) {
        requestAnimationFrame(tick);
      } else {
        const avgFPS = readings.reduce((a, b) => a + b, 0) / readings.length;
        const minFPS = Math.min(...readings);
        resolve({ avgFPS: Math.round(avgFPS), minFPS, readings });
      }
    }
    requestAnimationFrame(tick);
  });
})();
```

FPS monitoring procedure:
1. Navigate to page via `Playwright_navigate`
2. Inject FPS monitoring script
3. Scroll through entire page slowly (trigger all scroll-driven animations)
4. Collect FPS readings every 500ms over 3 seconds per scroll segment
5. Classify results:
```
Min FPS < 30        → CRITICAL FAIL (animation too heavy)
Avg FPS 30-45       → WARNING (concerning, may affect user experience)
Avg FPS 45+         → PASS
```

Important caveat to include in skill: FPS readings during Playwright-driven scroll may differ from real user scrolling. Flag < 30fps as a concern for investigation, not absolute truth. Note "FPS measured during automated scroll" in report.

**Layer 3: Integration Context**
- Quality-reviewer agent loads this skill and executes the protocol at end-of-build
- Testing runs in PARALLEL with CD review (LOCKED DECISION) -- CD reviews creative quality while quality-reviewer runs technical tests. Both produce findings.
- Results flow into the quality-gate protocol (04-05 skill) for severity classification
- Screenshots are saved to `.planning/modulo/audit/` for user review at Layer 4 checkpoint
- Build-orchestrator uses the overall verdict to decide: CRITICAL_FAIL = block + escalate, WARNINGS_ONLY = add to tally + checkpoint if accumulated, PASS = auto-proceed

**Layer 4: Anti-Patterns**
- Running full test suite per wave or per section (kills velocity -- run ONCE at end of build)
- Testing before polish pass (tests incomplete output -- polish may fix issues)
- Treating FPS monitoring as absolute truth (it's a signal, not precise measurement during automated scroll)
- Ignoring graceful degradation (Playwright MCP may not be configured in every user's environment)
- Hard-failing on every accessibility issue (only "critical" impact = hard fail, "serious" = warning, rest = info)
- Skipping screenshot comparison (screenshots without comparison against PLAN.md expectations are just pretty pictures, not quality verification)
- Testing only at 1440px (the most common problems are at 375px mobile)
