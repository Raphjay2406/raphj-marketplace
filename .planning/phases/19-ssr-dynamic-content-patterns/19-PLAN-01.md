**Layer 1: Decision Guidance must include:**

1. **When to Use** -- Any Genorah project that goes beyond fully static pages. Specific triggers:
   - Content updates more frequently than weekly (needs ISR or on-demand revalidation, not full rebuilds)
   - Pages mix public and personalized content (needs Cache Components or Server Islands)
   - CMS-driven content that editors publish without developer deploys (needs webhook revalidation)
   - Authenticated sections (dashboards, profiles, member pricing) coexist with public pages
   - Pages have slow data sources (needs streaming SSR with Suspense boundaries)
   - Database-driven dynamic routes with frequent data changes
   - Real-time or near-real-time data display (live scores, stock prices, social feeds)

2. **When NOT to Use** -- Redirect to other skills:
   - **Fully static marketing sites** -- Standard SSG (Next.js `generateStaticParams`, Astro default static) is sufficient; this skill adds unnecessary complexity
   - **Client-side data fetching (SWR, TanStack Query)** -- Deferred to v2. This skill covers server-side rendering strategies only
   - **API endpoint creation** -- Use `api-patterns` skill. This skill covers how PAGES render, not how API routes work
   - **Authentication UX (login, signup, sessions)** -- Use `auth-ui` skill. This skill covers how auth state affects RENDERING decisions, not auth flows themselves
   - **CMS data modeling or content fetching** -- Use `cms-integration` skill. This skill covers the REVALIDATION pipeline after CMS content changes, not how to fetch CMS data

3. **Rendering Strategy Decision Matrix** -- The core decision framework. Present as a 4-dimension matrix:

   **Dimensions:**
   - **Data Freshness**: Static (build-time only) | Stale-OK (minutes to hours) | Near-real-time (seconds) | Real-time (per-request)
   - **Personalization**: None | Segment-based (A/B, locale) | User-specific (auth, role)
   - **Build Frequency**: Rarely (deploy-triggered) | Daily (scheduled) | Continuous (CMS-driven)
   - **Runtime**: Node.js only | Edge-compatible | Static hosting

   **Decision Logic Table:**
   Map dimension combinations to rendering strategies:
   - No personalization + static data + any runtime = **SSG** (`generateStaticParams` / Astro default static)
   - No personalization + stale-OK + Node.js = **Cache Components** (`"use cache"` + `cacheLife`)
   - Segment personalization + stale-OK + Node.js = **Cache Components** with pass-through composition (segment as cache key argument)
   - User-specific + mostly static page + Node.js = **Cache Components shell + Suspense** for auth content (streams per-request)
   - Real-time + user-specific + Node.js = **Full SSR** (no cache, Suspense boundaries for slow data)
   - Any + Edge = **SSR without Cache Components** (Edge does NOT support `"use cache"` or ISR)
   - Astro equivalent: Static default + `server:defer` for dynamic parts; `prerender = false` for full SSR pages

4. **Named Scenario Recipes** -- 8-10 practical scenarios. For each: scenario name, typical data profile (freshness/personalization/build/runtime), recommended strategy for Next.js and Astro, and one key implementation note. Scenarios to include:

   1. **Blog / CMS Editorial** -- Stale-OK, no personalization, CMS-driven. Next.js: Cache Components + cacheLife('hours') + webhook revalidation. Astro: static build + ISR via Cache-Control headers.
   2. **E-commerce Product Page (PDP)** -- Stale-OK for product info, user-specific for pricing/cart. Next.js: Cache Components shell (product) + Suspense (pricing). Astro: static product + Server Island for cart/pricing.
   3. **Dashboard / Admin Panel** -- Real-time, user-specific, continuous. Next.js: Full SSR with Suspense boundaries, no cache. Astro: full SSR (`prerender = false`).
   4. **Marketing Landing Page** -- Static, no personalization, rarely updated. Next.js: SSG with `generateStaticParams`. Astro: default static.
   5. **User Profile / Settings** -- Per-request, user-specific. Next.js: Full SSR, auth check in Server Component. Astro: full SSR with Server Islands.
   6. **API-Driven Feed (News, Social)** -- Near-real-time, no personalization. Next.js: Cache Components with cacheLife('minutes'). Astro: SSR + short Cache-Control.
   7. **SaaS Pricing Page** -- Stale-OK, segment-based (locale/plan). Next.js: Cache Components with locale/segment as argument. Astro: static per-locale with Server Island for plan-specific pricing.
   8. **Documentation Site** -- Static, rarely updated, deploy-triggered. Next.js: SSG. Astro: default static (Astro is ideal for docs).
   9. **Marketplace Listing** -- Stale-OK (product), near-real-time (availability). Next.js: Cache Components (product info) + Suspense (live stock). Astro: static product + Server Island for stock.
   10. **Event / Conference Page** -- Near-real-time (schedule), CMS-driven. Next.js: Cache Components + cacheLife('minutes') + webhook. Astro: hybrid with SSR schedule component.

5. **Next.js 16 Cache System Breakdown** -- Explain the 4 cache layers clearly. This is a known developer pain point and must be thorough:

   1. **Request Memoization** -- Deduplicates identical fetch calls within a single render pass. Automatic, per-request, no configuration needed. Scope: one render.
   2. **Data Cache** -- Persistent server-side storage for cached data. Survives server restarts and deploys. Cleared by `revalidateTag(tag, 'max')` or `updateTag(tag)`. This is what `"use cache"` + `cacheLife` controls.
   3. **Full Route Cache** -- Pre-rendered HTML and RSC payload stored on the server. Depends on Data Cache; when Data Cache invalidates, Full Route Cache also invalidates for that route.
   4. **Router Cache** -- Client-side cache of visited routes. Survives navigation between pages. Has a minimum 30-second stale time. Cleared only by `router.refresh()`, hard refresh, or page reload. Key insight: invalidating server caches does NOT clear Router Cache -- the user may still see stale content for up to 30 seconds.

   Include a brief diagram (ASCII or markdown) showing the relationship: Request Memoization -> Data Cache -> Full Route Cache (server-side) | Router Cache (client-side, independent).

   **Invalidation API summary:**
   - `revalidateTag(tag, 'max')` -- Stale-while-revalidate: marks Data Cache entry as stale, serves stale on next request while regenerating in background. Use in Route Handlers (webhook endpoints).
   - `updateTag(tag)` -- Immediate expiry: removes Data Cache entry. Use in Server Actions for read-your-own-writes.
   - `revalidatePath(path)` -- Path-based invalidation. Simpler alternative when tags are not used.
   - `router.refresh()` -- Client-side: clears Router Cache for the current route. Call after server-side invalidation if immediate client freshness is needed.

6. **Edge vs Node.js Runtime Constraints** -- Present as a compact table from 19-RESEARCH.md. Key decision rule: if the page uses Cache Components, `"use cache"`, on-demand revalidation, `proxy.ts`, or direct database connections, it MUST use Node.js runtime. Edge is only for simple Route Handlers doing fetch-based proxying or redirects.

7. **Streaming SSR Concept** -- Brief explanation (5-8 sentences):
   - Streaming SSR sends the HTML shell immediately while slow data sources resolve
   - Uses React Suspense boundaries: `<Suspense fallback={<Skeleton />}><SlowComponent /></Suspense>`
   - The shell (everything outside Suspense) arrives instantly, improving TTFB
   - Each Suspense boundary streams its resolved content as it becomes available
   - Place Suspense boundaries around data-fetching components, NOT around static UI
   - Astro streams by default in SSR mode; for static pages, use Server Islands (`server:defer`) instead of Suspense
   - Key tradeoff: more Suspense boundaries = faster initial paint but more layout shifts

8. **`cacheLife` Preset Profiles** -- Reference table of the 7 built-in cacheLife profiles from 19-RESEARCH.md (default, seconds, minutes, hours, days, weeks, max) with stale/revalidate/expire values and use case. Also show the custom profile syntax in next.config.ts.

   - Output to: Page components with appropriate cache directives, webhook route handlers, next.config.ts cache configuration, Astro page files with prerender/SSR settings
