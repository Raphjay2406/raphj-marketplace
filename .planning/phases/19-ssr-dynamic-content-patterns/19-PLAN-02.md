**Layer 2B Part 2: Auth-Gated Content Rendering**

Add section header: `### Auth-Gated Content Rendering`

Brief intro (2-3 sentences): Auth-gated content requires careful rendering strategy decisions. The wrong approach either exposes private data in cached HTML, shows unnecessary loading spinners for content that could be pre-rendered, or breaks caching for the entire page. This section covers 3 mixed-page approaches and 4 auth library patterns.

**Three Mixed-Page Approaches:**

First, document the 3 approaches with trade-offs (as a comparison table or narrative):

1. **Server-Side Conditional Rendering** -- Check auth in the Server Component, render different content based on session state. Pros: no loading spinner, full SEO for public content, no client JS. Cons: cannot cache the page (every request hits the server). Best for: fully personalized pages (dashboards, profiles).

2. **Client-Side Auth Gate with Skeleton** -- Pre-render the public shell, use a client component that checks auth and conditionally renders private content. Pros: public shell is cached/static, private content loads asynchronously. Cons: layout shift when auth resolves, private content not in initial HTML (worse for perceived performance).

3. **Cache Components + Auth Boundary (Recommended for mixed pages)** -- Use `"use cache"` for the public content (cached for all users), wrap auth-dependent content in a Suspense boundary that streams per-request. Pros: best of both worlds (cached shell + streaming auth content), no layout shift if skeleton matches final layout, SEO-friendly public content. Cons: requires Node.js runtime, slightly more complex component architecture.

Show the Cache Components + Auth Boundary as the recommended approach using the product page example from 19-RESEARCH.md Pattern 5 (ProductInfo cached, PersonalizedPricing streamed per-request).

**Auth Library Server Component Patterns:**

For each of the 4 auth libraries, show the complete pattern for checking auth in a Server Component. Use 19-RESEARCH.md examples as source.

7. **Auth.js v5 (NextAuth v5)** -- Show: (a) the root `auth.ts` config with exported `auth()` function, (b) a protected Server Component using `const session = await auth()` with redirect on null session. Note: `auth()` works in Server Components, Route Handlers, proxy.ts, and Server Actions.

8. **Clerk** -- Show: a protected Server Component using `const { userId, isAuthenticated } = await auth()` and `const user = await currentUser()` from `@clerk/nextjs/server`. Note: `clerkMiddleware()` (now in `proxy.ts`) can protect entire route groups.

9. **Supabase** -- Show: (a) the `createClient()` helper using `@supabase/ssr` with cookie-based auth, (b) a protected Server Component using `supabase.auth.getClaims()`. CRITICAL: use `getClaims()` not `getSession()` for server-side validation. `getSession()` does NOT validate the JWT signature and can be spoofed.

10. **Better Auth** -- Show: a Server Component pattern using Better Auth's server-side session helper. Note: Better Auth is the modern replacement for Lucia (deprecated March 2025). Mark as MEDIUM confidence -- the API is newer and may evolve. Auth.js v5 and Clerk are higher-confidence choices for production projects.

**proxy.ts Route Protection Patterns:**

11. **Route Protection via proxy.ts** -- Show a Next.js 16 proxy.ts file that protects route groups (e.g., `/dashboard/*`) by checking auth and redirecting unauthenticated users. CRITICAL: the file is `proxy.ts` (NOT `middleware.ts` -- renamed in Next.js 16). Show the pattern generically and note which auth libraries provide their own proxy wrapper:
   - Auth.js v5: `auth` can be used as a proxy wrapper
   - Clerk: `clerkMiddleware()` with `createRouteMatcher` for protected routes
   - Supabase: `updateSession()` call in proxy to refresh auth cookies
   - Better Auth: Manual session check in proxy

**Role-Based Rendering:**

Pattern: check session -> extract role -> render appropriate component tree.
