**Layer 1: Decision Guidance** (~60-80 lines):

1. **Animation Performance Hierarchy:**
   ```
   Tier 0: CSS-only (compositor thread, 0 KB JS)
     - CSS transitions (Tailwind classes)
     - CSS @keyframes
     - CSS scroll-driven animations (@supports)

   Tier 1: Lightweight JS (~15-25 KB gzipped)
     - Motion library (tree-shaken with LazyMotion + domAnimation)
     - Intersection Observer (native, 0 KB added)

   Tier 2: Medium JS (~25-40 KB gzipped)
     - GSAP core (~25 KB)
     - GSAP + ScrollTrigger + SplitText (~40 KB)

   Tier 3: Heavy JS (~150+ KB gzipped) -- ALWAYS code-split
     - React Three Fiber + Three.js (~150 KB+)
     - Rive runtime (~100 KB)
     - Complex GSAP with all plugins (~60 KB)
   ```

   Rule: Always use the lowest tier that achieves the desired effect. Never import Tier 2+ on initial page load unless the animation is above the fold.

2. **CWV Impact Assessment:**
   | CWV Metric | Animation Risk | Mitigation |
   | LCP | JS-heavy animations blocking render | Code-split, defer non-critical animations |
   | INP | Animation JS blocking main thread | Use compositor-thread CSS, requestAnimationFrame |
   | CLS | Elements shifting during animation setup | Reserve layout space, use transform (not width/height) |

3. **When to Code-Split (mandatory):**
   - ANY import of Three.js, React Three Fiber, @react-three/drei
   - GSAP with 3+ plugins
   - Rive runtime (@rive-app/react-canvas)
   - Motion library when only used in one section (not site-wide)
   - dotLottie player (@lottiefiles/dotlottie-react)
   - Rule: If a library is used in only 1-2 sections, it MUST be dynamically imported

**Layer 2: Award-Winning Examples** (~180-230 lines):

1. **CSS Compositor-Thread Patterns** (~30 lines):
   Why CSS scroll-driven animations are the default:
   - Run on compositor thread (separate from main thread)
   - Zero JS overhead
   - Cannot cause jank even during heavy JS execution
   - Browser-optimized scheduling

   Properties that run on compositor: `transform`, `opacity`, `filter`, `clip-path`
   Properties that DON'T: `width`, `height`, `top`, `left`, `margin`, `padding`, `background-color`

   Show `@supports (animation-timeline: view())` pattern with graceful fallback (element visible, no animation in unsupported browsers).

2. **Code-Splitting Patterns** (~50 lines):

   Dynamic import for GSAP (only when section enters viewport):
   ```tsx
   'use client'
   import { useEffect, useRef } from 'react'

   function HeavyAnimation() {
     const ref = useRef<HTMLDivElement>(null)

     useEffect(() => {
       const observer = new IntersectionObserver(async ([entry]) => {
         if (entry.isIntersecting) {
           const gsap = (await import('gsap')).default
           const { ScrollTrigger } = await import('gsap/ScrollTrigger')
           gsap.registerPlugin(ScrollTrigger)
           // Animation code here
           observer.disconnect()
         }
       }, { rootMargin: '200px' })

       if (ref.current) observer.observe(ref.current)
       return () => observer.disconnect()
     }, [])

     return <div ref={ref}>...</div>
   }
   ```

   LazyMotion for tree-shaken Motion library:
   ```tsx
   import { LazyMotion, domAnimation, m } from 'motion/react'

   // domAnimation: ~15KB (basic animations)
   // domMax: ~25KB (full feature set including layout animations)
   function App({ children }) {
     return (
       <LazyMotion features={domAnimation} strict>
         {children}
       </LazyMotion>
     )
   }
   // Use <m.div> instead of <motion.div> with LazyMotion
   ```

   Next.js dynamic import for React Three Fiber:
   ```tsx
   import dynamic from 'next/dynamic'

   const Scene3D = dynamic(() => import('@/components/Scene3D'), {
     ssr: false,
     loading: () => <div className="w-full h-[400px] bg-surface animate-pulse" />,
   })
   ```

3. **will-change Discipline** (~25 lines):
   NEVER apply `will-change` in CSS stylesheets permanently. Apply dynamically:
   ```tsx
   // Apply before animation, remove after
   function animateWithWillChange(element: HTMLElement) {
     element.style.willChange = 'transform, opacity'

     // Animation here (GSAP, CSS, etc.)

     // Remove after animation completes
     element.addEventListener('transitionend', () => {
       element.style.willChange = 'auto'
     }, { once: true })
   }
   ```

   Why: Each `will-change` creates a compositor layer consuming GPU memory. 20+ elements with permanent `will-change` can cause memory pressure and actually REDUCE performance.

   Exception: Elements that are ALWAYS animating (e.g., ambient background, loading spinner) can keep `will-change`.

4. **Font Loading Strategy** (~30 lines):
   Fonts are a critical performance concern because they affect LCP:

   Next.js: Use `next/font` (automatic subsetting, self-hosting, font-display: swap):
   ```tsx
   import { Inter, Playfair_Display } from 'next/font/google'
   const body = Inter({ subsets: ['latin'], variable: '--font-body' })
   const display = Playfair_Display({ subsets: ['latin'], variable: '--font-display' })
   ```

   Astro: Use `@fontsource` packages for self-hosting:
   ```astro
   ---
   import '@fontsource-variable/inter'
   import '@fontsource-variable/playfair-display'
   ---
   ```

   Rules:
   - Always use variable fonts when available (1 file, all weights)
   - Always self-host (no Google Fonts CDN -- extra DNS lookup)
   - Always set `font-display: swap` (text visible immediately, font swaps in)
   - Preload the display font if it's used above the fold
   - Maximum 3 font families per project (display, body, mono)

5. **Performance Budgets** (~30 lines):
   Concrete budgets per animation approach:

   | Approach | JS Budget | Load Strategy | Max Elements |
   | CSS only | 0 KB | Inline in stylesheet | Unlimited |
   | CSS scroll-driven | 0 KB | Inline in stylesheet | Unlimited |
   | Motion (LazyMotion) | 15 KB | Bundled (tree-shaken) | Unlimited |
   | Motion (full) | 25 KB | Bundled | Unlimited |
   | GSAP core | 25 KB | Code-split if section-only | 50+ per page OK |
   | GSAP + plugins | 40 KB | Code-split | 50+ per page OK |
   | R3F + Three.js | 150+ KB | ALWAYS code-split, ssr: false | 1 scene per page |
   | Rive | 100 KB | Code-split | 2-3 per page |
   | dotLottie | 50 KB | Code-split | 3-5 per page |

   Total animation JS budget per page: aim for < 80 KB gzipped (excluding code-split on-demand chunks).

6. **Reduced Motion as Performance Boost** (~15 lines):
   `prefers-reduced-motion` isn't just accessibility -- it's a performance optimization:
   ```css
   @media (prefers-reduced-motion: reduce) {
     *, *::before, *::after {
       animation-duration: 0.01ms !important;
       animation-iteration-count: 1 !important;
       transition-duration: 0.01ms !important;
       scroll-behavior: auto !important;
     }
   }
   ```
   Include in globals.css as a baseline. Individual components can override with more nuanced reduced-motion alternatives (e.g., fade instead of slide).

7. **Animation FPS Monitoring** (~20 lines):
   Development-time FPS monitoring pattern:
   ```tsx
   // Dev-only FPS counter
   if (process.env.NODE_ENV === 'development') {
     let lastTime = performance.now()
     let frames = 0
     function checkFPS() {
       frames++
       const now = performance.now()
       if (now - lastTime >= 1000) {
         const fps = Math.round(frames * 1000 / (now - lastTime))
         if (fps < 30) console.warn(`[Genorah] Low FPS detected: ${fps}fps`)
         frames = 0
         lastTime = now
       }
       requestAnimationFrame(checkFPS)
     }
     requestAnimationFrame(checkFPS)
   }
   ```
   Flag anything below 30fps. Target: 60fps for all animations.

**Layer 3: Integration Context** (~30-40 lines):
- **-> Cinematic Motion**: The CSS-first decision tree in Cinematic Motion IS a performance decision. This skill provides the WHY behind that default.
- **-> Wow Moments**: Tier 3 wow moments (WebGL, Rive, dotLottie) require code-splitting documented here. This skill provides the loading patterns.
- **-> Design System Scaffold**: The scaffold generates the globals.css reduced-motion baseline and font loading setup. This skill provides the rationale.
- **-> Quality Gates (Phase 4)**: Live browser testing checks Lighthouse scores and FPS. This skill provides the budgets those gates enforce.
- **-> Page Transitions**: View Transitions API is zero-JS overhead (browser-native). AnimatePresence adds Motion bundle. This skill informs the choice.

**Layer 4: Anti-Patterns** (~30-40 lines):
- **Anti-Pattern: Importing Everything** -- `import gsap from 'gsap'` at the top level of layout.tsx when only one section uses GSAP. Why bad: 25-40 KB added to every page load. Instead: dynamic import when section enters viewport.
- **Anti-Pattern: will-change on Everything** -- `will-change: transform` in global CSS for `.animate` class. Why bad: GPU memory explosion, actually reduces performance. Instead: apply/remove dynamically via JS.
- **Anti-Pattern: Google Fonts CDN** -- `<link href="fonts.googleapis.com">`. Why bad: extra DNS lookup, no subsetting control, privacy concerns. Instead: self-host via next/font or @fontsource.
- **Anti-Pattern: Layout-Triggering Animations** -- Animating `width`, `height`, `top`, `left`. Why bad: triggers layout recalculation, causes jank. Instead: use `transform` (translateX/Y for position, scale for size).
- **Anti-Pattern: Motion Without LazyMotion** -- Importing full `motion/react` when only using basic animations. Why bad: 25KB instead of 15KB. Instead: use `LazyMotion` + `domAnimation` + `m` component.
- **Anti-Pattern: Three.js Without SSR Disable** -- Importing React Three Fiber in a server component or without `ssr: false`. Why bad: crashes during SSR, adds to initial bundle. Instead: always dynamic import with `ssr: false`.
