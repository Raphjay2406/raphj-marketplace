**Layer 1: Decision Guidance** (~60-80 lines):

1. **Page Transition Decision Tree:**
   ```
   Is this an Astro project?
     YES -> Use View Transitions API (stable, zero-JS)
     NO -> Is this Next.js?
       YES -> Is the project risk-tolerant (experimental OK)?
         YES -> Use View Transitions API (experimental) with AnimatePresence fallback
         NO -> Use AnimatePresence (reliable, production-ready)
       NO -> Use AnimatePresence (works everywhere React works)

   Do you need shared element morphing between routes?
     YES -> Use Motion layoutId (FLIP animation)
     NO -> Standard page enter/exit is sufficient
   ```

2. **Framework Maturity Matrix:**
   | Framework | View Transitions API | AnimatePresence | Shared Elements |
   | Astro | Stable, built-in, zero-JS | Not applicable (not React) | Via view-transition-name CSS |
   | Next.js App Router | Experimental (15.2+), may change | Stable, recommended | Via Motion layoutId |
   | React/Vite SPA | Via `document.startViewTransition()` | Stable | Via Motion layoutId |

   IMPORTANT: Next.js docs explicitly state View Transitions are "experimental and subject to change, not recommended for production."

3. **Per-Archetype Transition Choreography Table:**
   Map all 19 archetypes to their default page transition style:
   | Archetype | Transition Style | Duration | Easing |
   Key examples:
   - Brutalist: Instant/cut (no transition, raw) | 0ms | none
   - Luxury/Fashion: Slow crossfade with scale | 600-800ms | ease-out
   - Kinetic: Directional slide (in direction of navigation) | 300-400ms | spring-like
   - Japanese Minimal: Gentle fade | 400-500ms | ease-in-out
   - Editorial: Vertical wipe/reveal | 400-500ms | ease-out
   - Neon Noir: Glitch/static transition | 300-500ms | custom
   - Swiss: Clean horizontal slide | 300ms | ease-out
   Cover all 19 archetypes.

4. **When NOT to Transition:**
   - First page load (no exit animation, only entrance)
   - Same-page anchor links (scroll, don't transition)
   - Modal/dialog open/close (use AnimatePresence locally, not page-level)
   - Back navigation should feel different from forward (reverse the direction)

**Layer 2: Award-Winning Examples** (~200-260 lines):

1. **View Transitions API -- Astro (Stable)** (~40 lines):
   Complete Astro implementation:
   - `<ViewTransitions />` component in layout
   - CSS `view-transition-name` for shared elements
   - `::view-transition-old(*)` and `::view-transition-new(*)` pseudo-elements for custom animation
   - `astro:before-swap` and `astro:after-swap` events for lifecycle hooks
   - Zero-JS: all transitions handled by browser

2. **View Transitions API -- Next.js (Experimental)** (~40 lines):
   Experimental Next.js implementation:
   - `experimental: { viewTransition: true }` in next.config.js
   - `import { ViewTransition } from 'react'` (React 19+)
   - `<ViewTransition>` wrapper for elements that participate in transitions
   - IMPORTANT: Add comment noting experimental status and recommending AnimatePresence fallback
   - Show how to feature-detect and fall back:
     ```tsx
     const supportsViewTransitions = typeof document !== 'undefined' && 'startViewTransition' in document
     ```

3. **Motion AnimatePresence** (~50 lines):
   Production-ready page transition pattern:
   - Layout wrapper with `<AnimatePresence mode="wait">` keyed by pathname
   - Import from `motion/react` (NOT `framer-motion`)
   - `mode="wait"`: exit completes before enter starts (most common)
   - `mode="popLayout"`: exiting element popped from layout flow (for overlapping transitions)
   - Complete implementation: layout.tsx wrapper, page.tsx with motion.div, exit/enter variants
   - Archetype-specific variant examples:
     - Luxury crossfade variant: `{ initial: { opacity: 0, scale: 0.98 }, animate: { opacity: 1, scale: 1 }, exit: { opacity: 0, scale: 1.02 } }`
     - Kinetic slide variant: `{ initial: { x: '100%' }, animate: { x: 0 }, exit: { x: '-100%' } }`

4. **Shared Element Transitions** (~40 lines):
   Using Motion `layoutId` for FLIP animations:
   - List page: image card with `layoutId={`hero-${id}`}`
   - Detail page: hero image with same `layoutId={`hero-${id}`}`
   - Motion automatically morphs position, size, and shape between routes
   - `layout="position"` for position-only transitions (avoids width/height stretch)
   - Performance note: limit shared elements to 2-3 per transition (more causes jank)

5. **GSAP Flip Plugin for Complex Transitions** (~30 lines):
   For transitions that need timeline control:
   - `Flip.getState()` to capture before state
   - DOM manipulation
   - `Flip.from(state, { ... })` to animate from previous state
   - Useful when AnimatePresence is insufficient (e.g., grid-to-detail with multiple morphing elements)

6. **Per-Archetype Choreography Examples** (~30 lines):
   3-4 fully implemented variants showing archetype personality in transitions:
   - Brutalist: No transition (instant content swap, maybe a flash of the accent color)
   - Luxury: Long crossfade with subtle scale, background color transitions smoothly
   - Neon Noir: Brief glitch effect (translate X/Y jitter for 100ms) before fade
   - Ethereal: Dissolve with blur (filter: blur() transitions from 10px to 0)

**Layer 3: Integration Context** (~30-40 lines):
- **-> Cinematic Motion**: Page transitions use the archetype's base easing and duration. The transition timing should feel consistent with in-page motion.
- **-> Design DNA**: Transition colors (overlay, background during transition) come from DNA tokens. No arbitrary colors during transitions.
- **-> Emotional Arc**: Page transitions set the emotional tone for what's coming. A transition INTO a page with a HOOK beat should build anticipation.
- **-> Multi-Page Architecture** (Phase 8): This skill provides the transition mechanics. Phase 8's multi-page skill will define WHEN and WHERE transitions are used in site architecture.
- **-> Performance-Aware Animation**: View Transitions API is browser-native (zero JS overhead). AnimatePresence adds Motion bundle. GSAP Flip adds GSAP bundle. Choose based on performance budget.

**Layer 4: Anti-Patterns** (~30-40 lines):
- **Anti-Pattern: View Transitions in Next.js Production** -- Using experimental View Transitions API in production Next.js without fallback. Why bad: API may change, breaks in older browsers. Instead: use AnimatePresence as primary, View Transitions as progressive enhancement.
- **Anti-Pattern: Slow Transitions** -- Page transitions exceeding 500ms (except Luxury/Ethereal archetypes). Why bad: users perceive delay as slowness. Instead: 200-400ms for most archetypes.
- **Anti-Pattern: Same Transition for Forward and Back** -- Sliding right on both forward and back navigation. Why bad: breaks spatial model. Instead: forward = slide left, back = slide right (or reverse whatever direction).
- **Anti-Pattern: Too Many Shared Elements** -- 5+ elements morphing simultaneously between pages. Why bad: performance nightmare, visual chaos. Instead: 1-3 shared elements maximum.
- **Anti-Pattern: framer-motion Import** -- Using `import { AnimatePresence } from 'framer-motion'`. Why bad: deprecated package. Instead: `import { AnimatePresence } from 'motion/react'`.
- **Anti-Pattern: Transition Without Purpose** -- Adding a page transition "because we can." Why bad: some archetypes (Brutalist) are better with instant cuts. Instead: respect the archetype choreography table.
