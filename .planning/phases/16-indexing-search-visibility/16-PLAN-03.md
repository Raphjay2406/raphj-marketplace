# Search Visibility: Webmaster Tools Submission Workflows

> Covers: Google Search Console, Bing Webmaster Tools, Yandex Webmaster, Naver Search Advisor
> All verification methods documented equally -- choose based on your access level

## Before You Begin

Prerequisites:
1. Site is deployed and publicly accessible (verification files must be reachable)
2. Sitemap is generated and accessible at a known URL (e.g., /sitemap.xml or /sitemap-index.xml)
3. robots.txt references the sitemap URL (via `Sitemap:` directive)
4. You have DNS access (for DNS verification methods) or deploy access (for file/meta tag methods)

## Google Search Console

### Step 1: Add Property

1. Go to https://search.google.com/search-console
2. Click "Add Property"
3. Choose property type:
   - **Domain property** (recommended): Covers all subdomains, protocols, and paths. Requires DNS verification.
   - **URL prefix property**: Covers exact URL prefix only (e.g., https://example.com). More verification options.

### Step 2: Verify Ownership

[Table: Method | Steps | Best For]

Document ALL 5 methods equally from 16-RESEARCH.md Section 6 (GSC):
1. **HTML file upload** -- Download google*.html, place in public/ or site root. Deploy. Click Verify.
2. **HTML meta tag** -- Copy meta tag, add to <head> of all pages. For Next.js: use `metadata.verification.google` in root layout. Deploy. Click Verify.
3. **DNS TXT record** -- Copy TXT record value, add to DNS provider. Wait for propagation (up to 72 hours). Click Verify.
4. **Google Analytics** -- Auto-detected if GA tracking code present on site. Click Verify.
5. **Google Tag Manager** -- Auto-detected if GTM container present on site. Click Verify.

Include the Next.js verification shortcut from 16-RESEARCH.md:
```tsx
// app/layout.tsx
export const metadata: Metadata = {
  verification: {
    google: 'your-google-verification-code',
  },
}
```

### Step 3: Submit Sitemap

1. Navigate to Sitemaps in the left sidebar
2. Enter your sitemap URL (e.g., sitemap.xml or sitemap-index.xml)
3. Click Submit
4. Wait for processing (may take hours to days for first submission)

### Step 4: Monitor

1. Check "Coverage" report for indexing errors
2. Check "Sitemaps" report for submission status (Success/Has errors/Couldn't fetch)
3. Set up email alerts: Settings > Email preferences > Enable all notifications
4. Common issues: 404 errors (broken URLs), "Excluded" pages (canonicalized or noindexed), "Discovered - currently not indexed" (Google chose not to index)

**Important:** Google sitemap ping was deprecated June 2023. The ONLY paths for Google are: sitemap reference in robots.txt + GSC submission.

**Layer 4: Anti-Patterns must include (7 anti-patterns):**

Source from 16-RESEARCH.md "Anti-Patterns to Avoid" and "Common Pitfalls" but rewrite in the skill's Layer 4 format (Name / What goes wrong / Instead).

1. **Assuming Google Supports IndexNow**
   **What goes wrong:** Developers implement IndexNow thinking it covers Google, then wonder why Google indexing is slow. New pages appear in Bing within minutes but take days/weeks in Google.
   **Instead:** Always implement the dual-path strategy: IndexNow for Bing/Yandex/Naver + sitemap/GSC submission for Google. The engine comparison matrix in Layer 1 makes this explicit.

2. **Resubmitting Unchanged URLs**
   **What goes wrong:** Every deployment submits ALL URLs to IndexNow, not just changed ones. Search engines rate-limit (HTTP 429) or flag as spam. Wastes API quota.
   **Instead:** Implement content-hash tracking. For Next.js: in-memory Map or KV store. For Astro SSG: use astro-indexnow with cache file. Only submit URLs where the content hash has changed.

3. **Blocking AI Search Bots When Blocking Training Bots**
   **What goes wrong:** Using blanket AI bot blocks (or blocking GPTBot to stop training) also kills visibility in ChatGPT search, Perplexity, DuckAssist. Site disappears from AI-powered search.
   **Instead:** Use the Selective preset: explicitly allow search bots (OAI-SearchBot, PerplexityBot, Claude-SearchBot, etc.) before blocking training bots (GPTBot, Google-Extended, ClaudeBot, etc.). See appendix-ai-crawlers.md.

4. **Missing IndexNow Key Verification File**
   **What goes wrong:** IndexNow submissions return 403 Forbidden. Key validation shows as pending (202 that never resolves).
   **Instead:** Place `public/{YOUR_KEY}.txt` containing only the key string. File name must match the key exactly. Verify it's accessible at `https://yoursite.com/{key}.txt` after deploy.

5. **Using Google Sitemap Ping (Deprecated)**
   **What goes wrong:** Code pings `google.com/ping?sitemap=...` which has returned 404 since June 2023. No harm, but no benefit -- and developers think Google is being notified.
   **Instead:** For Google: submit sitemap via GSC + reference in robots.txt. That's the only path. Remove any sitemap ping code.

6. **Overselling llms.txt Impact**
   **What goes wrong:** Over-investing in llms.txt optimization (spending hours on content, A/B testing descriptions) while basic SEO (meta tags, sitemaps, structured data) is incomplete. No measurable ranking impact because no AI platform has confirmed reading these files.
   **Instead:** Implement llms.txt as a low-effort, forward-looking convention. Spend 15-30 minutes max. Prioritize seo-meta and structured-data skills first.

7. **Submitting Sitemaps Before Verification**
   **What goes wrong:** Sitemap submission fails or is silently ignored because site ownership isn't verified. Developer thinks sitemap is submitted but it's not being processed.
   **Instead:** Follow the appendix-submission.md workflow: verify ownership FIRST, then submit sitemap. All four platforms require verified ownership before accepting sitemap submissions.

**Machine-Readable Constraints table:**

After Layer 4, add the enforceable constraint table. These are the search-visibility parameters quality-reviewer checks:

| Parameter | Min | Max | Unit | Enforcement |
|-----------|-----|-----|------|-------------|
| IndexNow batch size | 1 | 10000 | URLs per POST | HARD -- reject if exceeded |
| IndexNow key length | 8 | 128 | chars | HARD -- reject if outside range |
| IndexNow key characters | - | - | a-z, A-Z, 0-9, - only | HARD -- reject invalid chars |
| IndexNow key verification file | - | - | present at /{key}.txt | HARD -- reject if missing |
| IndexNow submission endpoint | - | - | api.indexnow.org | HARD -- reject per-engine endpoints |
| robots.txt AI preset selection | - | - | one of: open, selective, restrictive | HARD -- must choose a preset |
| robots.txt sitemap reference | - | - | present | HARD -- reject if missing |
| llms.txt H1 header | - | - | present (site name) | SOFT -- warn if missing |
| llms.txt file location | - | - | /llms.txt at site root | SOFT -- warn if wrong path |
| Webmaster tools verification | - | - | at least GSC + Bing verified | SOFT -- warn if unverified |
| Content-hash tracking | - | - | implemented for sites with 100+ URLs | SOFT -- warn if missing for large sites |
| Google sitemap ping | - | - | must NOT be present | HARD -- reject (deprecated June 2023) |

Note: HARD constraints cause build/review failures. SOFT constraints produce warnings. The distinction reflects that some items (IndexNow spec compliance) are protocol requirements while others (llms.txt, webmaster verification) are best practices.
