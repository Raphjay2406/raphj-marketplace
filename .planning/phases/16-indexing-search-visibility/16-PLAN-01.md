   a. **Honest Disclosure** (3-4 sentences): "IndexNow does NOT work for Google. Google tested it in 2022 but never adopted it. For Google, the only paths are: submit sitemap via GSC and reference it in robots.txt. IndexNow works for Bing, Yandex, Naver, Seznam, Yep, Internet Archive, and Amazon -- all via a single POST to api.indexnow.org."

   b. **Engine x Method Comparison Matrix** -- Copy from 16-RESEARCH.md Section 7 "Unified Indexing Strategy":
   | Engine | IndexNow | Sitemap | Webmaster Tools | Sitemap Ping | Notes |

   c. **Dual-Path Strategy Summary**: For every project: (1) Generate sitemap + submit to GSC for Google, (2) Set up IndexNow endpoint for everything else. Google sitemap ping was deprecated June 2023 -- skip entirely.

   d. **Project-Type Recipes** -- Copy from 16-RESEARCH.md Section 7:
   - Static blog / marketing site: sitemap on build + IndexNow on deploy via astro-indexnow or build script

   - SaaS landing page (few pages, infrequent changes): sitemap + GSC only, IndexNow optional

Use the "explain then recipe" structure: for each pattern, 1-3 sentences explaining the indexing principle, then the code.

**A. IndexNow Auto-Setup Patterns (IDX-01):**

1. **IndexNow API Key Setup** -- Brief explanation of the key requirements (8-128 hex chars, a-z/A-Z/0-9/hyphens only). Two approaches:
   - Generate via Bing Webmaster Tools dashboard (recommended for production)
   - Generate any valid hex string for development
   - Environment variable: `INDEXNOW_KEY` + `INDEXNOW_INTERNAL_SECRET` (for endpoint auth)
   - Framework config patterns: `next.config.ts` env section, Astro `astro.config.mjs` env section

2. **IndexNow Key Verification File** -- Explain: IndexNow requires a static file at `/{key}.txt` to verify domain ownership. Pattern from 16-RESEARCH.md:
   - Static approach: `public/{YOUR_API_KEY}.txt` containing only the key string (both frameworks)
   - Dynamic approach (Next.js only): Route handler at `app/[key]/route.ts` that serves the key. Use the exact pattern from 16-RESEARCH.md "IndexNow Key Verification File Endpoint."

3. **Next.js 16: IndexNow Route Handler** -- Use the complete pattern from 16-RESEARCH.md "Pattern 1." This is the primary pattern for Next.js projects. Must include:
   - Internal auth via Bearer token (INDEXNOW_INTERNAL_SECRET)
   - Content-hash tracking with in-memory Map (explain: persists across warm serverless invocations, lost on cold start -- acceptable for most sites; for persistent tracking, note KV store option)
   - SHA-256 content hashing via `createHash`
   - URL filtering: skip unchanged URLs
   - Batch submission in groups of 10,000 (IndexNow limit)
   - Single POST to `https://api.indexnow.org/IndexNow` (NOT per-engine endpoints)
   - Full request body: host, key, keyLocation, urlList
   - Response handling with status tracking
   Note: Use `import { createHash } from 'crypto'` (Node.js built-in, no external deps)

4. **Astro SSR: IndexNow API Endpoint** -- Use the pattern from 16-RESEARCH.md "Pattern 2." For Astro projects running in SSR/hybrid mode with dynamic content updates. Must include:
   - `APIRoute` type import
   - Internal auth check
   - POST to api.indexnow.org
   - `import.meta.env` for configuration (NOT process.env)
   Note: Simpler than Next.js pattern because SSR Astro sites typically have fewer dynamic URLs. Content-hash tracking can be added following the same Map pattern.

5. **Astro SSG: astro-indexnow Integration** -- Use the pattern from 16-RESEARCH.md "Pattern 3." For static Astro sites deployed via CI/CD. Must include:
   - `astro-indexnow` v2.1.0 integration config in `astro.config.mjs`
   - Explanation of built-in features: HTML output hashing, automatic batching, cache file (.astro-indexnow-cache.json)
   - CI/CD note: commit `.astro-indexnow-cache.json` to git for cross-build persistence
   - Limitation: build-time only, NOT suitable for SSR sites with runtime updates
   - Installation command: `npm install astro-indexnow`

6. **Content-Hash Tracking Approaches** -- A reference table explaining when to use each approach. Use the table from 16-RESEARCH.md Section 8:
   | Approach | Best For | Pros | Cons |
   In-memory Map (Next.js serverless), File-based JSON (Astro SSG CI/CD), KV store (high-traffic SSR), Database column (CMS-driven).
   Include the astro-indexnow reference implementation summary (5-step process: scan HTML, hash, compare cache, submit changed, update cache).

7. **IndexNow Monitoring Pattern** -- Brief monitoring section:
   - Log every submission: URL count, response status, timestamp
   - Track 429 responses: implement exponential backoff with Retry-After header
   - Best practice: minimum 5 minutes between resubmissions of same URL
   - Rate limit awareness: 10,000 URLs per POST (firm), daily limits per engine unpublished
