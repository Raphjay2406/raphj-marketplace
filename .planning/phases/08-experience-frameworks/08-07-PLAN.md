**Layer 1: Decision Guidance (~60 lines)**

Framework detection: `vite.config.ts` + no Next.js/Astro signals -> React/Vite.

React/Vite is a CLIENT-ONLY SPA. Key differences from Next.js:
- No server components (everything is client-rendered)
- No `next/image` (use standard `<img>` with `loading="lazy"`)
- No `next/font` (use CSS `@font-face` or Fontsource)
- No `next/link` (use react-router or TanStack Router)
- No SSR FOUC prevention (SPA loads all at once -- simpler dark mode script)
- No API routes (backend is separate or API-based)

Decision tree:
- Client-side routing -> React Router v7 (most common) or TanStack Router (type-safe)
- Font loading -> CSS `@font-face` in globals.css or Fontsource packages
- Image optimization -> `<img>` with `loading="lazy"`, `srcset`, `width`/`height`
- Dark mode -> Inline `<script>` in `index.html` `<head>` (simpler than SSR frameworks)
- Code splitting -> React.lazy + Suspense for route-based splitting

When to use: SPA projects without SSR needs, rapid prototypes, projects with separate backend APIs.
When NOT to use: Content-heavy sites needing SEO (use Next.js or Astro), desktop apps (use desktop-patterns skill alongside).

**Layer 1: Decision Guidance (~70 lines)**

Framework detection:
- `tauri.conf.json` or `src-tauri/` -> Tauri
- `electron-builder.json` or `main.js` with electron import -> Electron

Tauri vs Electron decision (Modulo doesn't choose -- project already has one):
- Tauri: Rust backend, smaller bundle, newer, uses system webview
- Electron: Node.js backend, mature ecosystem, bundles Chromium

Desktop-specific design concerns that web-only skills miss:
1. **Titlebar area**: App must account for custom or native titlebar space
2. **Drag regions**: User needs to drag window by clicking somewhere
3. **Window controls**: Minimize/maximize/close buttons (platform-specific position)
4. **Platform differences**: macOS traffic lights (top-left) vs Windows controls (top-right)
5. **Content offset**: Main content must NOT be hidden behind titlebar
6. **System tray**: Background functionality UI
7. **Multi-window**: Settings, preferences, floating panels
8. **Native menus**: macOS menu bar, context menus
9. **Keyboard shortcuts**: Cmd vs Ctrl platform awareness

Decision tree:
- Custom titlebar desired -> Use `titleBarStyle: 'hidden'` + custom HTML titlebar
- Native titlebar sufficient -> No custom UI needed, just account for height
- macOS + custom titlebar -> Preserve native traffic lights, add right-side controls
- Windows + custom titlebar -> Full custom controls (minimize, maximize, close)

When to use: Any Tauri or Electron project.
When NOT to use: Web-only projects.

**Layer 2: Award-Winning Examples (~180 lines)**

Pattern 1: **Tauri v2 Custom Titlebar Component**
Show complete React component with:
- `data-tauri-drag-region` on titlebar container
- Window control buttons (minimize, maximize, close) via `@tauri-apps/api/window`
- Platform-aware layout (macOS traffic lights left, controls right; Windows all right)
- DNA integration (bg-surface, text-muted, border-border)
- Accessibility: button labels, keyboard operable
```tsx
'use client'
import { appWindow } from '@tauri-apps/api/window'

function TitleBar() {
  return (
    <div data-tauri-drag-region className="fixed top-0 inset-x-0 h-8 flex items-center justify-between px-3 bg-surface border-b border-border select-none z-50">
      {/* ... platform-aware controls */}
    </div>
  )
}
```

Pattern 2: **Electron Custom Titlebar**
Show BrowserWindow config and CSS approach:
```javascript
const win = new BrowserWindow({
  titleBarStyle: 'hidden',
  titleBarOverlay: {
    color: '#0a0a0f',       // DNA bg color
    symbolColor: '#f0ece6', // DNA text color
    height: 40,
  },
})
```
CSS `app-region: drag` for custom titlebar, `app-region: no-drag` for interactive elements within.

Pattern 3: **Content Offset for Titlebar**
Show how main content must account for titlebar height:
```tsx
// Tauri: fixed titlebar with body padding
<body className="pt-8"> {/* titlebar height */}
  <TitleBar />
  <main>{children}</main>
</body>

// Electron with titleBarOverlay: account for env(titlebar-area-height)
<body style={{ paddingTop: 'env(titlebar-area-height, 40px)' }}>
```

Pattern 4: **Platform Detection**
```typescript
const isMacOS = navigator.platform.includes('Mac')
const isWindows = navigator.platform.includes('Win')
const isLinux = navigator.platform.includes('Linux')

// macOS: traffic lights are top-left, need left padding
// Windows/Linux: controls are top-right, need right padding
```

Pattern 5: **Multi-Window Patterns**
Show settings/preferences window pattern:
- Main window spawns secondary window
- Consistent DNA theming across all windows
- Window-specific layouts (settings: sidebar + content, floating: compact)

Pattern 6: **Desktop-Specific Responsive Considerations**
Desktop apps have different responsive needs:
- Minimum window size (not 375px mobile -- typically 800x600 minimum)
- Resizable panels (sidebar drag handles)
- No mobile breakpoints needed (but support narrow window states)
- Keyboard shortcuts instead of touch interactions

**Layer 3: Integration Context (~50 lines)**

DNA Connection: Titlebar, window controls, and system tray all use DNA tokens. The titlebar IS a design element and should match the project's visual identity.

Archetype Variants: Titlebar style varies by archetype:
- Brutalist: Thick border below titlebar, stark text
- Ethereal: Translucent titlebar, subtle separator
- Neon Noir: Dark titlebar with glow accent
- Swiss: Clean, minimal titlebar with precise alignment

Related Skills: tailwind-system (same @theme tokens), dark-light-mode (titlebar must update with theme), accessibility (keyboard shortcuts, window control labels), react-vite-patterns (desktop apps typically use React/Vite underneath).

**Layer 4: Anti-Patterns (~40 lines)**

Anti-pattern 1: **Web-only design in desktop shell** -- What: Ignoring titlebar, drag regions, native integration. Why wrong: Feels like a website in a frame. Fix: Full desktop-aware design.

Anti-pattern 2: **One-size-fits-all titlebar** -- What: Same titlebar on macOS and Windows. Why wrong: Traffic lights (left) vs controls (right) differ. Fix: Platform detection and adaptive layout.

Anti-pattern 3: **Forgetting content offset** -- What: Content hidden behind fixed titlebar. Why wrong: First content is invisible. Fix: Body padding matching titlebar height.

Anti-pattern 4: **Mobile breakpoints in desktop** -- What: 375px responsive patterns in desktop app. Why wrong: Desktop window minimum is typically 800px. Fix: Desktop-appropriate minimum widths.

Anti-pattern 5: **Drag region on interactive elements** -- What: Buttons inside drag region don't work. Why wrong: Drag region captures all clicks. Fix: Explicit `no-drag` on interactive children.
