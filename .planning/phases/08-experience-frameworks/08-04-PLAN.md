**Layer 1: Decision Guidance (~70 lines)**

Core philosophy: Both modes independently award-worthy. Neither derived from the other. Dark mode is not "invert."

DNA generates BOTH palettes during `/gen:start-design`:
- User defines primary mode's palette
- Archetype inversion rules derive the other mode (if user only defines one)
- Both palettes stored in DESIGN-DNA.md
- Both reviewed for contrast compliance and aesthetic quality

Token mapping: The same 12 token NAMES (bg, surface, text, border, primary, secondary, accent, muted, glow, tension, highlight, signature) map to different VALUES in each mode. CSS custom properties switch values, utility classes stay the same.

Decision tree:
- Toggle mechanism -> Class-based (`.dark` on `<html>`) via Tailwind v4 `@custom-variant dark`
- Persistence -> `localStorage` with `prefers-color-scheme` as fallback
- FOUC prevention -> Inline `<script>` in `<head>`, before any rendering
- Framework integration -> next-themes (Next.js), inline `is:inline` script (Astro), inline in `index.html` (React/Vite)
- Transition animation -> Per-archetype signature transition (see Pattern 3)
- Asset variants -> CSS `light-dark()` function for simple color swaps, class-based `dark:` for structural changes
- Shadow system -> Light mode uses opacity-based shadows, dark mode uses colored glow shadows

When NOT to use: Only skip if project explicitly declares single-mode-only (rare).

**Layer 2: Award-Winning Examples (~220 lines)**

Pattern 1: **Complete Tailwind v4 Dark Mode Setup**
Show the full CSS setup (reference RESEARCH.md Pattern 3):
```css
@import "tailwindcss";
@custom-variant dark (&:where(.dark, .dark *));

@theme {
  --color-bg: #faf9f6;
  --color-surface: #ffffff;
  --color-text: #1a1a1a;
  /* ... all 12 tokens for light */

  --shadow-subtle: 0 1px 2px rgba(0,0,0,0.04);
  --shadow-elevated: 0 4px 12px rgba(0,0,0,0.06);
  --shadow-float: 0 8px 30px rgba(0,0,0,0.08);
}

.dark {
  --color-bg: #0a0a0f;
  --color-surface: #111115;
  --color-text: #f0ece6;
  /* ... all 12 tokens for dark, INDEPENDENTLY DESIGNED */

  --shadow-subtle: 0 1px 2px rgba(0,0,0,0.3);
  --shadow-elevated: 0 4px 12px rgba(0,0,0,0.4);
  --shadow-float: 0 8px 30px rgba(0,0,0,0.5), 0 0 20px var(--color-glow);
}
```

Pattern 2: **FOUC Prevention Per Framework**
Show inline `<script>` for `<head>` that runs before rendering:
- Core script: Check localStorage, fallback to prefers-color-scheme, apply .dark class
- Next.js: Use `next-themes` library (handles FOUC, persistence, system preference, SSR)
- Astro: Inline script in base layout with `is:inline` directive
- React/Vite: Inline script in `index.html` `<head>`

Pattern 3: **Archetype Signature Transitions**
The theme toggle moment is a DESIGN element. Show per-archetype transitions:
- Brutalist: Instant hard-cut (no animation, matches harsh aesthetic)
- Ethereal: Light-bloom fade (radial white flash fading to new theme, 800ms)
- Kinetic: Wipe transition (diagonal or circular wipe, 400ms)
- Neon Noir: Glow pulse (border glow flashes then settles to new theme, 500ms)
- Swiss/International: Clean crossfade (simple opacity, 300ms)
- Japanese Minimal: Zen dissolve (very slow crossfade, 1000ms)
- Glassmorphism: Frosted glass spread (blur effect radiating from toggle button)
- Luxury/Fashion: Elegant curtain (vertical wipe with slight ease, 600ms)
- (Brief entries for remaining archetypes)

Implementation approaches:
- CSS-only: `view-transition-name` API (native browser, progressive enhancement)
- Fallback: CSS transitions on custom properties + class toggle
- Complex: Motion library for archetype-specific animations (only when CSS insufficient)

Show View Transitions API approach (Baseline 2024 for same-document):
```tsx
function toggleTheme() {
  if (!document.startViewTransition) {
    document.documentElement.classList.toggle('dark')
    return
  }
  document.startViewTransition(() => {
    document.documentElement.classList.toggle('dark')
  })
}
```

With archetype-specific `::view-transition-*` CSS rules for the transition animation.

Pattern 4: **Dual Asset Support**
How to handle images, logos, illustrations that need different versions:
- CSS `light-dark()` function for background images (Baseline 2024)
- `<picture>` element with `prefers-color-scheme` media query for `<img>`
- Class-based `dark:hidden` / `dark:block` for showing/hiding alternate versions
- Photos: automatic treatment CSS filter (dark mode: reduced brightness, slight desaturation)
```css
.dark img:not([data-theme-aware]) {
  filter: brightness(0.85) saturate(0.9);
}
```

Pattern 5: **Dark Mode Design Differences (Not Just Color Swaps)**
Beyond colors, what changes between modes:
- Shadows: Light uses subtle opacity, dark uses colored glows
- Borders: Light uses subtle dark borders, dark uses subtle light borders
- Depth: Light mode uses elevation (shadow), dark mode uses luminance (surface brightness)
- Images: Dark mode photos slightly dimmed, illustrations use dark variants
- Gradients: Different gradient intensities and directions
- Text weight: Body text may appear thinner in dark mode -- consider `font-weight` adjustment

**Layer 3: Integration Context (~60 lines)**

DNA Connection: DESIGN-DNA.md stores both palettes. The `colors` section has `light` and `dark` sub-objects. DNA generation (Phase 1) validates both palettes for contrast compliance.

Archetype Variants: Each archetype defines:
1. Signature transition animation (see Pattern 3)
2. Dark mode depth treatment (glow intensity, surface differentiation)
3. Whether certain archetype elements change between modes (e.g., Neon Noir's glow is more prominent in dark)

Pipeline Stage: Design System Scaffold generates both palette sets from DNA. Section builders use DNA tokens that automatically switch. The toggle component is typically part of navigation (Wave 1).

Related Skills: tailwind-system (dark mode CSS architecture, @custom-variant), design-dna (palette generation), accessibility (contrast compliance in BOTH modes), responsive-design (toggle must be accessible at all sizes).

**Layer 4: Anti-Patterns (~50 lines)**

Anti-pattern 1: **Palette inversion** -- What: `--dark-bg: invert(--light-bg)`. Why wrong: Produces washed-out, unprofessional dark modes. Fix: Independently design both palettes.

Anti-pattern 2: **FOUC** -- What: No inline script, dark mode flashes on load. Why wrong: Jarring user experience. Fix: Inline `<script>` in `<head>` before any rendering.

Anti-pattern 3: **Missing dark mode shadow rethink** -- What: Same shadow values in both modes. Why wrong: Shadows are invisible against dark backgrounds. Fix: Dark mode uses colored glows, light mode uses opacity shadows.

Anti-pattern 4: **v3 darkMode config** -- What: `darkMode: 'class'` in tailwind.config.ts. Why wrong: v4 uses CSS `@custom-variant dark`. Fix: CSS-first dark mode.

Anti-pattern 5: **Ignoring image treatment** -- What: Same bright images in dark mode. Why wrong: Bright images are jarring in dark UI. Fix: Auto-dim photos, provide dark-variant illustrations.

Anti-pattern 6: **No transition animation** -- What: Instant toggle with no visual feedback. Why wrong: Misses design opportunity. Fix: Archetype-signature transition animation.
