**Layer 1: Decision Guidance (~80 lines)**

Core philosophy: WCAG 2.1 AA is the FLOOR, not the ceiling. Every component, every interaction, every state must be accessible. No exceptions.

This skill defines STANDARDS. Other skills IMPLEMENT them. When writing any component pattern in any skill, it must include:
1. Semantic HTML (correct elements, not div-soup)
2. ARIA attributes where semantics are insufficient
3. Keyboard navigation (tab order, arrow keys for groups)
4. Focus indicators (archetype-styled, not generic blue outlines)
5. `motion-safe:` / `motion-reduce:` variants on all animations
6. Color contrast compliance (4.5:1 body text, 3:1 large text, 3:1 UI components)
7. 44px minimum touch targets on mobile

Decision tree:
- Interactive custom widget -> Full ARIA roles, states, properties + keyboard handler
- Standard HTML element (`<button>`, `<a>`, `<input>`) -> Semantic HTML sufficient, no ARIA needed
- Decorative element -> `aria-hidden="true"`, `role="presentation"`
- Dynamic content update -> `aria-live` region (polite for non-urgent, assertive for alerts)
- Hidden content for screen readers -> `sr-only` Tailwind class (preferred) or `aria-label`
- Modal/dialog -> Focus trap, `role="dialog"`, `aria-modal="true"`, `inert` on background content
- Visually hidden text -> Use `sr-only` class, NOT `display:none` (which hides from screen readers too)

When NOT to use: Never. Accessibility is always active.

Pattern 2: **Keyboard Navigation Patterns**
Show complete keyboard handling for:
- Tab navigation: Natural tab order following DOM, `tabindex="0"` only when needed, never `tabindex` > 0
- Arrow key navigation within groups (roving tabindex pattern):
  ```tsx
  // Toolbar with roving tabindex
  function Toolbar({ items }) {
    const [activeIndex, setActiveIndex] = useState(0)
    const handleKeyDown = (e, index) => {
      if (e.key === 'ArrowRight') setActiveIndex((index + 1) % items.length)
      if (e.key === 'ArrowLeft') setActiveIndex((index - 1 + items.length) % items.length)
      if (e.key === 'Home') setActiveIndex(0)
      if (e.key === 'End') setActiveIndex(items.length - 1)
    }
    return (
      <div role="toolbar" aria-label="Actions">
        {items.map((item, i) => (
          <button
            key={item.id}
            tabIndex={i === activeIndex ? 0 : -1}
            onKeyDown={(e) => handleKeyDown(e, i)}
            ref={i === activeIndex ? (el) => el?.focus() : undefined}
          >{item.label}</button>
        ))}
      </div>
    )
  }
  ```
- Focus trap for modals (using `inert` attribute on background, Baseline 2023):
  ```tsx
  // When modal opens: document.querySelector('main').setAttribute('inert', '')
  // When modal closes: document.querySelector('main').removeAttribute('inert')
  ```
- Skip links (jump past navigation to main content):
  ```html
  <a href="#main-content" class="sr-only focus:not-sr-only focus:absolute focus:top-4 focus:left-4 focus:z-50 focus:px-4 focus:py-2 focus:bg-primary focus:text-bg focus:rounded">
    Skip to main content
  </a>
  ```

Pattern 3: **ARIA Patterns for Common Components**
Show correct ARIA for:
- Accordion: `role="region"`, `aria-expanded`, `aria-controls`, `aria-labelledby`
- Tabs: `role="tablist"`, `role="tab"`, `role="tabpanel"`, `aria-selected`, arrow key nav
- Dropdown menu: `role="menu"`, `role="menuitem"`, `aria-expanded`, `aria-haspopup`
- Toast/notification: `role="status"`, `aria-live="polite"` (or `role="alert"` for errors)
- Modal: `role="dialog"`, `aria-modal="true"`, `aria-labelledby`, focus trap, escape to close
- Carousel: `role="region"`, `aria-roledescription="carousel"`, `aria-label`, live region for slide changes

Pattern 5: **Color Accessibility**
- Contrast ratios: 4.5:1 for body text, 3:1 for large text (18px+ or 14px+ bold), 3:1 for UI components/borders
- Color-blind safe: never rely on color alone for information. Use icons, labels, patterns alongside color
- Each archetype's palette should include notes on which color combinations meet/fail contrast
- Testing: recommend axe-core for automated checks (catches ~57% of WCAG issues), manual verification for the rest

**Layer 4: Anti-Patterns (~60 lines)**

Anti-pattern 1: **Div-soup** -- What: `<div onClick={...}>Click me</div>`. Why wrong: Not focusable, no keyboard interaction, no role. Fix: Use `<button>` for actions, `<a>` for navigation.

Anti-pattern 2: **Visible but inaccessible** -- What: Custom dropdown with no ARIA. Why wrong: Screen readers can't announce state changes. Fix: Full ARIA roles, states, properties.

Anti-pattern 3: **Generic focus indicators** -- What: Default blue outline or `outline: none`. Why wrong: Either ugly (blue) or invisible (none). Fix: Archetype-styled focus indicators via `:focus-visible`.

Anti-pattern 4: **Animation without motion-reduce** -- What: `animate-bounce` with no reduced-motion alternative. Why wrong: Can cause vestibular issues. Fix: `motion-safe:animate-bounce motion-reduce:animate-none`.

Anti-pattern 5: **Color-only information** -- What: Red border for errors, green for success. Why wrong: Color-blind users can't distinguish. Fix: Icons + labels + color.

Anti-pattern 6: **tabindex > 0** -- What: `tabindex="5"` to force tab order. Why wrong: Breaks natural flow, creates maintenance nightmare. Fix: Use DOM order for tab sequence.

Anti-pattern 7: **aria-label on everything** -- What: Adding aria-label to elements that already have visible text. Why wrong: Overwrites visible text for screen readers, causing confusion. Fix: Use aria-label only when visible text is insufficient or absent.
