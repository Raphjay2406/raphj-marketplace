  <action>
Rewrite `commands/iterate.md` as a brainstorm-first design change command (~100-140 lines). Structure:

**Frontmatter:**
```yaml
---
description: Improve designs with brainstorm-first approach -- 2-3 proposals with mockups before any changes
argument-hint: [description of desired change or section name]
allowed-tools: Read, Write, Edit, Bash, Grep, Glob, Task
---
```

**Body structure:**

1. **Role statement**: "You are the Genorah Iterate orchestrator. You improve existing designs through a mandatory brainstorm-first process -- presenting 2-3 distinct approaches with ASCII mockups before touching any code. Every iteration is informed, not impulsive."

2. **Guided Flow Header**: Read STATE.md and CONTEXT.md. Display one-line status.

3. **State Check & Auto-Recovery**:
   - Required state: Any state with built sections (EXECUTION_COMPLETE, or wave in progress with completed sections)
   - If no built sections: "Nothing to iterate on yet. Run /gen:execute first."
   - Check for GAP-FIX.md files (from quality review). If they exist, offer: "Quality review found [N] gaps. Fix those first? Or proceed with your iteration request?"

4. **Argument Parsing**:
   - Bare text = description of desired change
   - `--section name` or `-s name` = target specific section
   - `--plan-only` or `-po` = create iteration plan without executing (for review)
   - `--from-gaps` or `-g` = execute existing GAP-FIX.md plans from quality review
   - Image path = analyze screenshot to understand what to change

5. **Scope Determination**:
   - If --from-gaps: read GAP-FIX.md files and present them. Skip brainstorm (gaps are already diagnosed).
   - If $ARGUMENTS has description: use as starting point for brainstorm.
   - If no arguments: ask user what they want to change and in which sections.

6. **Brainstorm Phase (MANDATORY)** -- This is the core differentiator:
   - Dispatch to `creative-director` agent with: change request, current section code, DESIGN-DNA.md, adjacent section info
   - Creative-director generates 2-3 distinct approaches. Each MUST include:
     ```
     ### Approach [N]: [Descriptive Name]

     **What changes:**
     - [specific change 1]
     - [specific change 2]

     **ASCII Mockup (Desktop 1440px):**
     [ASCII art showing the layout change -- not optional, MANDATORY]

     **Blast radius:**
     - This section: [what changes]
     - Adjacent above ([section name]): [impact or "none"]
     - Adjacent below ([section name]): [impact or "none"]
     - Shared components: [impact or "none"]

     **Pros:** [benefits]
     **Cons:** [tradeoffs]
     ```
   - Present all approaches to user. Wait for selection.
   - If user says "approach 2": proceed with that approach.
   - If user suggests a different direction: brainstorm again with new input.
   - If user says "combine 1 and 3": creative-director merges approaches.
   - Approaches MAY reference third-party marketplace components (Aceternity UI, Magic UI, 21st.dev) when relevant. The actual marketplace knowledge skill is Phase 7, but brainstorm proposals can suggest them as options.
   - NEVER skip this step, even for "obvious" changes. The brainstorm catches unintended consequences.

7. **Implementation**:
   - Once user selects an approach, dispatch to `polisher` or `section-builder` agent
   - Agent applies the chosen approach with exact changes
   - If blast radius flagged adjacent sections: ask user before touching them
   - Atomic commits per section: `refactor(section-XX): description of improvement`

8. **Post-Change Verification**:
   - Dispatch to `quality-reviewer` for targeted check on changed sections only
   - Report any issues. If new gaps found, create GAP-FIX.md.

9. **Completion & Next Step**:
   ```
   Iteration complete.

   Changed: [section names]
   Approach: [chosen approach name]
   Commits: [list of commits]

   Next step: /gen:iterate (for more changes)
     Or: /gen:execute --resume (if mid-build)
     Or: /gen:audit (for comprehensive review)
   ```

10. **Rules** (5-7 items):
    - ALWAYS brainstorm before implementing. No exceptions. Even for "just change the color."
    - Each approach MUST include an ASCII mockup. Descriptions alone are insufficient.
    - Check blast radius on adjacent sections. Smart adjacency: flag ripple effects, ask before touching.
    - User selects the approach. The command never auto-selects.
    - Minimal changes. Iterate improves, it doesn't redesign (unless explicitly requested).
    - Preserve the creative direction. Iterations refine within the archetype, not against it.
    - Always end with a clear next step.

**Critical constraints:**
- Target 100-140 lines. Brainstorm generation logic lives in creative-director agent.
- The brainstorm gate is NON-NEGOTIABLE. It's the entire point of this command vs raw editing.
- ASCII mockups in proposals are MANDATORY per CONTEXT.md decision.
- change-plan.md functionality (plan modifications) is absorbed here -- iterate can also modify plans.

  <name>Task 2: Write bug-fix.md command</name>
  <files>commands/bug-fix.md</files>
  <action>
Create `commands/bug-fix.md` as a diagnostic brainstorm command (~90-120 lines). Structure:

**Frontmatter:**
```yaml
---
description: Diagnose and fix visual bugs with root cause analysis before any changes
argument-hint: [bug description or path to screenshot]
allowed-tools: Read, Write, Edit, Bash, Grep, Glob, Task
---
```

**Body structure:**

1. **Role statement**: "You are the Genorah Bug-Fix orchestrator. You diagnose visual bugs using a systematic hypothesis-test cycle -- brainstorming the CAUSE, not the solution. No fix is applied without understanding the root cause first."

2. **Guided Flow Header**: Read STATE.md and CONTEXT.md. Display one-line status.

3. **State Check & Auto-Recovery**:
   - Required state: Any state with built sections
   - If no built sections: "Nothing to fix yet. Run /gen:execute first."

4. **Argument Parsing**:
   - Bare text = bug description
   - Image/screenshot path = analyze the screenshot to identify the visual issue
   - `--section name` or `-s name` = target specific section
   - `--viewport size` or `-v size` = viewport where bug appears (mobile, tablet, desktop)
   - No arguments: ask user to describe the bug

5. **Bug Understanding**:
   - If screenshot provided: analyze it and describe what's wrong
   - Read the affected section's PLAN.md (what it SHOULD look like)
   - Read the actual implementation (what it IS)
   - Classify the bug using visual audit categories (spacing, alignment, typography, color, responsive, interactive, borders, icons, animation, accessibility)

6. **Diagnostic Brainstorm (MANDATORY)** -- This is fundamentally different from iterate's creative brainstorm:
   - Dispatch to `quality-reviewer` agent with: bug description, section code, PLAN.md, DESIGN-DNA.md
   - Quality-reviewer investigates and presents diagnostic:
     ```
     "This could be caused by:
     A) [hypothesis 1] -- because [evidence from code]
     B) [hypothesis 2] -- because [evidence from code]
     C) [hypothesis 3] -- because [evidence from code]

     Let me investigate..."
     ```
   - Quality-reviewer tests each hypothesis against the code
   - Present confirmed root cause + proposed fix:
     ```
     Root Cause: [confirmed hypothesis]
     Evidence: [specific file:line and what's wrong]

     Fix:
     - File: [path]
     - Change: [old] -> [new]
     - Why: [explanation]

     Risk: [adjacent section impact or "None"]

     Apply this fix?
     ```
   - Wait for user approval.

7. **Fix Application**:
   - Dispatch to `polisher` agent to apply the minimal fix
   - Minimal changes only: fix the bug, don't refactor or improve
   - Verify no regression in adjacent sections
   - Atomic commit: `fix(section-XX): description of what was fixed`

8. **Post-Fix Report**:
   ```
   Bug Fix Report

   Bug: [description]
   Root Cause: [cause]
   Fix: [what was changed]
   Files: [modified files]
   Regression: None / [description]
   ```

9. **Completion & Next Step**:
   ```
   Bug fixed.

   Next step: /gen:bug-fix (if more bugs)
     Or: /gen:iterate (for design improvements)
     Or: /gen:execute --resume (if mid-build)
   ```

10. **Rules** (5-6 items):
    - Diagnose before fixing. NEVER apply a fix without confirmed root cause.
    - Minimal changes only. Fix the bug, don't redesign the section.
    - One bug at a time. Sequential diagnosis prevents confusion.
    - Check for regressions after every fix.
    - Atomic commits. Each bug fix gets its own commit.
    - If the bug reveals a plan gap, note it but don't modify the plan -- suggest /gen:iterate.

**Critical difference from iterate:**
- Iterate brainstorms CREATIVE APPROACHES (2-3 mockups, user picks)
- Bug-fix brainstorms the ROOT CAUSE (hypotheses, tests, confirmed diagnosis)
- Iterate explores "what could this become?"
- Bug-fix investigates "what went wrong?"
