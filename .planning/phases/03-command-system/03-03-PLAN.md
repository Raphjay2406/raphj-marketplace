Create `commands/plan-dev.md` as a thin router command (~100-140 lines). Structure:

**Frontmatter:**
```yaml
---
description: Create detailed build plans for each section with wave assignments and context-rot-safe chunks
argument-hint: [--phase N] [--skip-research] [--section name]
allowed-tools: Read, Write, Edit, Bash, Grep, Glob, Task
---
```

**Body structure:**

1. **Role statement**: "You are the Genorah Plan-Dev orchestrator. You create section-level build plans that are context-rot-safe -- each plan is self-contained enough for a builder agent to execute without needing the full project context."

2. **Guided Flow Header**: Read STATE.md and CONTEXT.md. Display one-line status.

3. **State Check & Auto-Recovery**:
   - Required state: CONTENT_COMPLETE or DNA_COMPLETE
   - Auto-recovery matrix:
     * No STATE.md at all -> "Run /gen:start-project first."
     * STATE.md exists, no DNA -> "Creative direction needed. Running start-project DNA generation..."
     * DNA exists, no content and --skip-content not set -> Offer to run content planning or skip
   - Check for DISCUSSION-{phase}.md. If none exists, auto-offer:
     "No creative discussion for this phase yet. Want to deep-dive on creative features first? (/gen:lets-discuss) Or proceed directly to planning?"
     Wait for user response. If they proceed, continue without discussion.

4. **Argument Parsing**:
   - `--phase N` or `-p N` = plan specific phase
   - `--skip-research` or `-sr` = skip phase-scoped re-research
   - `--section name` or `-s name` = plan single section only
   - `--dry-run` or `-d` = show proposed sections and waves without creating PLAN.md files

5. **Phase 1: Re-Research** (skip if --skip-research):
   - Spawn `researcher` agent scoped to the current phase's sector (e.g., if planning a pricing phase, research pricing page patterns specifically)
   - Researcher updates research findings focused on this phase's needs
   - Brief: present key findings relevant to the phase. Don't dump the full research.

6. **Phase 2: Section Identification & Wave Assignment**:
   - Dispatch to `section-planner` agent with: PROJECT.md, BRAINSTORM.md, DESIGN-DNA.md, CONTENT.md, DISCUSSION-{phase}.md (if exists), and re-research findings
   - Section-planner identifies all sections, assigns waves, emotional beats, wow moments, creative tensions, and transition techniques
   - Present section map to user for approval:
     ```
     ## Section Plan Overview

     | # | Section | Wave | Beat | Wow Moment | Tension |
     |---|---------|------|------|------------|---------|
     | 00 | shared | 0 | -- | -- | -- |
     | 01 | hero | 2 | HOOK | gradient-mesh | scale-violence |
     ...

     [N] sections across [M] waves. Approve this structure?
     ```
   - Wait for user approval. If changes requested, adjust and re-present.

7. **Phase 3: PLAN.md Generation**:
   - Section-planner generates PLAN.md per section in `.planning/genorah/sections/XX-{name}/PLAN.md`
   - Each PLAN.md has GSD frontmatter (section, wave, depends_on, files_modified, autonomous, must_haves)
   - Each PLAN.md has context-rot-safe chunks: visual specification, component structure, wow moment code, creative tension spec, exact copy from CONTENT.md
   - Plans include verification questions that builders answer before executing (canary-style DNA recall)
   - Present each section plan to user for approval (per-section, not batch)

8. **Phase 4: Master Plan & State Update**:
   - Create/update MASTER-PLAN.md with wave map, dependency graph, file structure
   - Update STATE.md: phase -> PLANNING_COMPLETE
   - Update CONTEXT.md with section table, beat sequence, wave map

9. **Completion & Next Step**:
   ```
   Planning complete.

   [N] section plans created across [M] waves.
   Master plan: .planning/genorah/MASTER-PLAN.md

   Next step: /gen:execute
     Builds sections wave by wave with parallel builders.
     Or: /gen:execute --dry-run to preview the build order first.
   ```

10. **Rules** (5-8 items):
    - Every section needs a PLAN.md with GSD frontmatter. No exceptions.
    - Wave assignments must respect dependencies. Max 4 sections per wave.
    - Each section plan requires individual user approval. Never batch-approve.
    - Include at least one checkpoint:human-verify per section.
    - Plans must be self-contained: a builder should need only PLAN.md + DESIGN-DNA.md.
    - All domain logic (beat assignment, wow moment selection, tension placement) lives in section-planner agent.
    - If DISCUSSION-{phase}.md exists, its decisions take priority over default creative choices.
    - Always end with a clear next step.

**Critical constraints:**
- Target 100-140 lines. Section planning logic lives in section-planner agent.
- The command's unique value: re-research, discussion integration, user approval flow, and state management.
- DO NOT embed ASCII art templates, beat type lists, wow moment catalogs, or tension type lists. Those belong in agents/skills.
- The lets-discuss auto-offer is a key differentiator from v6.1.0.
