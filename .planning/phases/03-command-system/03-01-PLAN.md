  <action>
Create `commands/start-project.md` as a thin router command (~120-150 lines). Use this exact structure:

**Frontmatter:**
```yaml
---
description: Start a new Modulo project -- discovery, research, creative direction, and content planning
argument-hint: [project description or path to requirements file]
allowed-tools: Read, Write, Edit, Bash, Grep, Glob, Task, WebFetch
---
```

**Body structure (follow the Command Template from research):**

1. **Role statement** (1-2 lines): "You are the Modulo Start-Project orchestrator. You guide users through discovery, research, and creative foundation -- feeling like a conversation with a creative director, not a form."

2. **Guided Flow Header section**: Read `.planning/modulo/STATE.md` and `.planning/modulo/CONTEXT.md`. Display one-line status. If a project already exists, warn: "A Modulo project already exists. Starting over will replace existing artifacts. Continue?"

3. **State Check & Auto-Recovery section**: If STATE.md already exists with a completed start-project phase, offer to skip to plan-dev. If no state exists, proceed normally. This is the first command so auto-recovery is minimal.

4. **Argument Parsing section**: Parse `$ARGUMENTS` for:
   - Bare text = project description
   - File path (.md, .txt, .pdf) = read requirements from file
   - Image path = analyze as visual reference
   - `--skip-content` or `-sc` = skip content planning phase (defer to later)
   - `--minimal` or `-m` = minimal discovery (fewer questions)

5. **Phase 1: Discovery section** -- Adaptive questioning (this is in-command, not dispatched):
   - Step 1: Check for attached files from $ARGUMENTS
   - Step 2: Essential batch (4-5 questions presented conversationally, NOT as a numbered form):
     "Tell me about your project. I need to understand:
     - What is the product or service?
     - Who is your target audience?
     - What feeling should the site convey? (2-3 adjectives)
     - Any reference sites you admire? (URLs or descriptions work)"
   - Step 3: Conversational follow-ups based on interesting answers (provide 4-5 example follow-up triggers: luxury -> ask what luxury means, competitors -> ask what they get right, minimal answers -> ask about hard no's, specific features -> probe the user journey)
   - Step 4: Soft approval -- present condensed brief, suggest proceeding. NOT a formal approval gate.
   - Important: explicitly state "Do NOT present questions as a numbered form. This should feel like a creative director conversation."

6. **Phase 2: Research section** -- Dispatch to agents:
   - Spawn 4 parallel `researcher` agents via Task tool, each with a research track:
     (a) Design trends for the industry/tone
     (b) Reference site analysis (if URLs provided)
     (c) Component/pattern library research
     (d) Animation/motion technique research
   - Include instruction for live progress streaming: "As each researcher completes, report findings to the user in real-time. Show what was discovered as it comes in."
   - After all researchers complete, synthesize findings and present to user.
   - Mention once that user can share reference URLs/screenshots, but don't push.

7. **Phase 3: Creative Direction section** -- Dispatch to creative-director agent:
   - Spawn `creative-director` agent with research output + discovery answers
   - Creative-director selects archetype, generates creative direction, produces BRAINSTORM.md
   - Present 1 strong recommendation as a visual showcase:
     * Condensed "design brief" with archetype personality summary
     * Key color descriptions (not hex dumps)
     * Typography personality (not font names list)
     * Signature element description
   - Escape hatch: "If this doesn't feel right, I have alternative directions from the research. Or tell me what to change and I'll regenerate."
   - Soft approval gate: suggest next step naturally. No formal "approve" button.

8. **Phase 3.5: DNA Generation section**:
   - Creative-director generates DESIGN-DNA.md from chosen direction
   - Initialize CONTEXT.md with DNA identity anchor
   - Present DNA summary to user (brief, not the full document)

9. **Phase 4: Content Planning section** (skip if --skip-content):
   - Spawn `content-specialist` agent with PROJECT.md + BRAINSTORM.md + DESIGN-DNA.md
   - Content-specialist generates CONTENT.md with all page copy
   - Present content for user approval: "Content is as important as visual design. Please review."
   - User must approve content before proceeding.

10. **State Initialization section**:
    - Create/update STATE.md with phase: CONTENT_COMPLETE (or DNA_COMPLETE if --skip-content)
    - Initialize CONTEXT.md if not already done

11. **Completion & Next Step section**:
    ```
    Start-Project complete.

    Artifacts created:
      .planning/modulo/PROJECT.md      -- Requirements
      .planning/modulo/BRAINSTORM.md   -- Creative direction: [archetype]
      .planning/modulo/DESIGN-DNA.md   -- Visual identity
      .planning/modulo/CONTENT.md      -- Approved page copy
      .planning/modulo/CONTEXT.md      -- Context anchor
      .planning/modulo/STATE.md        -- Project state

    Next step: /modulo:plan-dev
      Creates detailed build plans for each section with wave assignments.
      Or: /modulo:lets-discuss to deep-dive on specific creative features first.
    ```

12. **Rules section** (5-8 items):
    - Never skip discovery. Understanding prevents rework.
    - Research before brainstorming. Directions must be informed by real design intelligence.
    - Present 1 strong recommendation, not 3 equal options. Have an opinion.
    - If the user rejects the direction, show alternatives before asking what to change.
    - Discovery should feel like talking to a creative director, not filling out a form.
    - All domain logic (archetypes, scoring, DNA format) belongs in agents, not this command.
    - Always end with a clear next step.
    - Track state. Update STATE.md after each phase transition.

**Critical constraints:**
- Target 120-150 lines total. Domain knowledge stays in agents.
- Use the `Task` tool references for agent spawning (not literal Task tool calls -- this is a prompt that Claude will follow).
- Match the frontmatter format of existing v6.1.0 commands (description, argument-hint, allowed-tools).
- Do NOT include archetype lists, scoring rubrics, DNA templates, or ASCII art templates in the command. Those belong in skills/agents.
