Rewrite `commands/execute.md` as a thin wrapper (~80-120 lines). This is the thinnest command because all execution logic lives in the build-orchestrator agent.

**Frontmatter:**
```yaml
---
description: Build sections wave by wave with parallel builders
argument-hint: [--wave N] [--resume] [--dry-run] [--parallel N]
allowed-tools: Read, Write, Edit, Bash, Grep, Glob, Task
---
```

**Body structure:**

1. **Role statement**: "You are the Modulo Execute orchestrator. You dispatch to the build-orchestrator agent which manages wave-based parallel execution, session boundaries, and quality gates."

2. **Guided Flow Header**: Read STATE.md and CONTEXT.md. Display one-line status:
   `Phase: Build | Wave: [X/Y] | Sections: [built/total] | Next: [current action]`

3. **State Check & Auto-Recovery**:
   - Required state: PLANNING_COMPLETE or later (or existing wave in progress for resume)
   - Auto-recovery matrix:
     * No STATE.md -> "Run /modulo:start-project first."
     * STATE.md exists, no plans -> "No section plans found. Running plan-dev first..." then auto-chain to plan-dev, then execute.
     * Plans exist, all sections COMPLETE -> "All waves complete. Run /modulo:iterate to refine, or /modulo:audit for comprehensive review."
   - Transparent auto-recovery: always tell user what's being run automatically.

4. **Argument Parsing**:
   - `--wave N` or `-w N` = start from specific wave number
   - `--resume` or `-r` = resume from CONTEXT.md state
   - `--dry-run` or `-d` = show execution plan without building
   - `--parallel N` or `-p N` = limit parallel builders (default: 4, max: 4)
   - Bare word "resume" = same as --resume (backward compatibility)
   - Auto-detection: if CONTEXT.md exists with incomplete state and no flags, treat as --resume

5. **Session Resume Boot Sequence** (if --resume or auto-detected):
   - Read CONTEXT.md (single file has everything: DNA identity, build state, next instructions)
   - Read next wave's section PLAN.md files (paths from CONTEXT.md)
   - Run canary check: "Before continuing, verify from memory: Display font? Accent-1 hex? Forbidden patterns? Layouts used so far? Next beat type?"
   - If canary check fails (2+ wrong): recommend new session
   - Present wave summary to user before continuing

6. **Dry Run Mode** (if --dry-run):
   - Display the full wave map from MASTER-PLAN.md
   - Show which sections are pending, which are complete
   - Show estimated parallel builders per wave
   - Do NOT execute anything. End with: "Run /modulo:execute to start building."

7. **Execution Dispatch**:
   - Dispatch to `build-orchestrator` agent with:
     * Starting wave number (from flags or auto-detected)
     * Parallel limit (from --parallel or default 4)
     * MASTER-PLAN.md path
     * DESIGN-DNA.md path
     * CONTEXT.md path
   - The build-orchestrator handles ALL execution logic:
     * Wave ordering and dependency checking
     * Parallel builder spawning via Task tool
     * Post-wave canary checks and CONTEXT.md rewriting
     * Creative Director review per wave
     * Session boundary management (2-wave soft suggestion)
     * Layout diversity enforcement
     * Real-time status updates

8. **Post-Execution Handling**:
   - Build-orchestrator returns control after:
     (a) All waves complete -> show completion summary
     (b) Session boundary reached -> show session save message
     (c) Error/failure -> show error and suggest /modulo:bug-fix
   - Update STATE.md based on build-orchestrator's report

9. **Completion & Next Step** (varies by exit condition):
   - All complete:
     ```
     All [N] waves complete. [X] sections built.

     Next step: /modulo:iterate
       Review the build and request design improvements.
       Or: /modulo:audit for comprehensive quality review.
     ```
   - Session boundary:
     ```
     Session state saved to CONTEXT.md.
     [N] waves complete, [M] remaining.

     Next step: /modulo:execute --resume (in a new session)
       Picks up exactly where you left off.
     ```
   - Error:
     ```
     Build paused due to error in [section].
     Error: [brief description]

     Next step: /modulo:bug-fix [section description]
       Diagnose and fix the issue, then resume with /modulo:execute --resume.
     ```

10. **Rules** (5-6 items):
    - This command is a thin wrapper. All execution logic lives in build-orchestrator.
    - Always read CONTEXT.md first when resuming. It has everything.
    - Never skip the canary check on resume. Context rot is the #1 quality killer.
    - Transparent auto-recovery: tell the user what's being run automatically.
    - Build failures bubble to the user. The orchestrator does NOT auto-retry.
    - Always end with a clear next step.

**Critical constraints:**
- Target 80-120 lines. This should be the shortest core command.
- All wave management, builder spawning, diversity tracking, and session management logic moves to build-orchestrator agent.
- The command handles: state check, flags, dispatch, and post-execution messaging. Nothing else.
- Do NOT include the layout pattern types list, wave execution steps, or builder spawn details. Those are build-orchestrator's domain.
